Design a NoSQL Store from scratch

Not "NoSQL means no schema and it scales." Instead: two Principal Architects trading precise, checkable arguments in a design review — where nothing is allowed to be a definition, and every mechanism has to earn its place by defeating the decision before it.

Stage 0

Don't say "NoSQL"

Architect A

I don't want NoSQL. You already run Postgres for mutual fund orders and it works fine. Convince me a relational database genuinely isn't enough — before you name a single product.

✎ think first

If you're storing NSDL demat holdings, insurance policy documents, and IPO allotment records — three wildly different shapes — what does forcing them into one rigid relational schema actually cost you, mechanically, not philosophically?

Reveal the dialoguereveal
B

A mutual fund order is five flat fields. An insurance policy has nested riders, beneficiaries, a payment schedule. An IPO allotment record has a different shape again. Force all three into fixed columns and you either drown in nullable columns or fragment each record across a dozen joined tables.

RIGID SCHEMA:                     REALITY:
  orders(id, amt, ...)               order: 5 flat fields
  policies(id, amt, ...,             policy: nested riders,
    rider1, rider2, rider3...)               beneficiaries, schedule
  -- most rider columns sit NULL     allotment: yet another shape
     for every order row
A

So store JSON blobs in a JSONB column in Postgres. Why do you need an entirely new system for that?

B

At modest scale, you don't — JSONB genuinely solves the shape problem. The real wall isn't schema flexibility at all. It's write throughput on a single node.

A

Prove it. Where, mechanically, does one node actually stop?

B
SINGLE NODE, EVERY WRITE:
  client --> write --> WAL append (fsync to disk) --> commit
  Every writer queues behind the SAME disk, the SAME WAL, the SAME lock.

Add more application servers -- doesn't matter.
They all funnel into ONE physical write path.

Read replicas can absorb read load forever. Not one of them can take a single write off that one physical WAL. That ceiling is fixed by disk fsync latency, not by schema design.

Where this leaves us: the real problem was never "schema flexibility" — that was a convenient story. The actual, undeniable wall is horizontal write scaling. Everything from here on is a consequence of accepting that this is the problem we're actually solving.
Stage 1

What are we optimizing?

Architect A

Pick exactly one: strong consistency, write scalability, or query flexibility. Not all three. Defend the one you pick from the disk up.

✎ think first

If writes are spread across many independent nodes to escape the single-WAL ceiling from stage 0, what do you lose the instant a query needs to touch two of those nodes at once?

Reveal the dialoguereveal
B

Write scalability. The single-WAL ceiling from stage 0 is fixed by physics — one disk, one append log — and the only way past a physical ceiling is to stop relying on one physical thing.

A

What does that cost you, precisely — not vaguely?

B

Cross-record transactions. The instant a customer's mutual fund orders and their NSDL holdings live on different nodes, an ACID join across both needs distributed coordination — locks held across a network, not just across memory. That coordination is exactly the kind of shared choke point we just spent stage 0 trying to eliminate.

SINGLE NODE:                        MULTIPLE NODES:
  join = read two tables,              join = ask Node A, ask Node B,
  same disk, same transaction          coordinate across a NETWORK,
  -- cheap, fast, atomic                agree on ONE consistent answer
                                        -- expensive, and can fail mid-way
A

So you're deliberately giving up the thing relational databases are best at.

B

Deliberately, and with eyes open — because the workload here is millions of independent writes per key, not complex multi-table joins. We're not choosing NoSQL because joins are unfashionable; we're choosing it because the join isn't the expensive operation we actually need to be fast at.

Where this leaves us: optimizing for write scale forces a partition-first architecture — data has to be split across independent nodes before anything else is designed. Every later stage is a direct consequence of having already accepted that trade.
Engineering Philosophy

We're not choosing NoSQL because joins are unfashionable. We're choosing it because the join isn't the expensive operation we need to be fast at. Always know what you're giving up, and why the tradeoff is worth it for your specific workload.

Stage 2

Spread the data randomly. Watch reads collapse.

Architect A

Simplest possible split: every write goes round-robin to whichever node is next. No key, no logic. What breaks?

✎ think first

If a single customer's records are scattered across every node at random, what does fetching "everything for this customer" actually require?

Reveal the dialoguereveal
B

Reads become scatter-gather. Fetching one customer's holdings means asking every single node "do you have any of this customer's records?" and waiting for the slowest one to answer before you can return anything.

Customer C's records, scattered randomly:
  Node1: [C's order]        Node2: [ ]           Node3: [C's holding]
  Node4: [ ]                Node5: [C's premium]  ... 50 more nodes

  Query "give me everything for C" --> ask ALL 54 nodes
  --> answer is only as fast as the SLOWEST one to respond
A

Doesn't matter how fast your writes got, then, does it — you've just made every read slower by adding nodes.

B

Right — random placement solved write distribution and quietly broke read locality, which is the exact opposite of what a data store is for. The fix is that related records need to live together, on purpose, not scattered by coincidence.

Where this leaves us: "spread the writes" was never the whole design — it has to be "spread the writes while keeping each customer's own data on one node." That single requirement is what forces the next decision: a deliberate partition key, not a coin flip.
Stage 3

Choosing the partition key — and watching it betray you

Architect A

Partition by customer ID range — customers 1 to 1,000,000 on Node 1, the next million on Node 2, and so on. Clean, predictable. What's wrong with it?

✎ think first

Customer IDs are assigned sequentially as people onboard. Where does every single new customer's data physically land, forever?

Reveal the dialoguereveal
B

Every new customer lands on whichever node holds the highest ID range — always the same node, forever, because IDs only ever go up. That node takes 100% of new-account write traffic while every other node sits idle holding old, cold data.

RANGE PARTITION BY SEQUENTIAL ID:
  Node1: IDs 1-1M       (old, cold, quiet)
  Node2: IDs 1M-2M      (old, cold, quiet)
  Node3: IDs 2M-3M+     (every NEW customer lands HERE, forever)

  100% of onboarding write traffic --> one node. Always the same one.
A

Fine — hash the ID instead. Problem solved?

B

Hashing spreads writes uniformly — hash(customer_id) mod N lands each customer on a pseudo-random node, so new customers spread evenly, not onto one hot node. But it destroys range queries. "Give me every customer who onboarded this week" used to be one sequential scan; now those customers are scattered across every node by the hash.

RANGE (sequential IDs):             HASH (uniform spread):
  Node1: [1..1000]                    Node1: [4, 891, 12002...]
  scan range = ONE node, sequential   Node1: same query = scatter-gather
                                       across every node again
A

So you just reintroduced stage 2's problem for a different query pattern.

B

For that specific pattern, yes — and that's the honest trade, not a flaw I can engineer away. Hashing is the right default because write-hotspotting from sequential IDs is the more damaging, more common failure; range scans on onboarding date can be served from a separate, purpose-built index instead of forcing the primary partition key to do both jobs.

Where this leaves us: the partition key decides who suffers — either new writes concentrate on one node, or range scans scatter across all of them. There's no key that avoids both; there's only a key that decides which pain the system is built to tolerate. This exact trade resurfaces in stage 11, once real-world data stops being evenly sized.
Engineering Philosophy

There is no partition key that is perfect for all queries. Pick the key that makes your most critical query fast and accept that every other query pays a cost. That's a real design decision, not a failure.

Stage 4

Replication — and who's allowed to write

Architect A

Why replicate at all? Don't say "availability." Everyone says that. Give me the mechanism.

✎ think first

If a single partition holding one customer's entire NSDL demat holding lives on exactly one disk, what does "that disk fails" actually mean for that data?

Reveal the dialoguereveal
B

Disks fail. Not rarely — routinely, at scale. If a partition holding a customer's demat holdings has exactly one copy and that disk dies, the data isn't "temporarily unavailable." It's gone. For a financial record, that's not a degraded experience, it's an unrecoverable loss.

A

So replicate three copies. Do all three accept writes, or one?

B

One leader per partition accepts writes; the other two replicate it. If all three accepted writes independently, a brief network split could let two of them diverge — two different "truths" about the same customer's holding, with no obvious way to say which one is correct.

ONE LEADER PER PARTITION:            ALL THREE WRITABLE:
  Leader:  [write] -> [write]           Replica1: [A][B]     <- diverged
  Follower: copies leader's order       Replica2: [A][C]     <- during a
  Follower: copies leader's order       Replica3: [A][B]        network split
A

What did you just give up by insisting on one leader?

B

Write availability during a leader failure — until a new leader is elected, that partition can't accept writes at all. That gap is bounded and measurable. Divergent history, on the other hand, is not bounded — it can silently corrupt the record forever.

Where this leaves us: single-leader replication trades a short, honest pause for a guarantee against silent corruption. That's a strong stance — but it only holds as long as the network between nodes behaves. Stage 5 is where we ask what happens the moment it doesn't.
Stage 5

The trade-off you cannot engineer around

Architect A

The network between your leader and its two followers splits — a genuine partition, not a metaphor. A client asks the follower side for data. Answer immediately with something possibly stale, or refuse to answer until the network heals. Pick one. There is no third option.

✎ think first

Name a real record type in this platform where a stale-but-available answer is actively dangerous — and one where it's perfectly fine. What separates them?

Reveal the dialoguereveal
B

There genuinely is no third option — that's not a design failure, it's a proof. During a real network partition, a node is physically unable to both confirm it has the newest data and respond instantly; it can only do one.

NETWORK PARTITION, follower side asked for data:

  CHOOSE CONSISTENCY:        CHOOSE AVAILABILITY:
    "I might be stale.         "Here's my last known value,
     I refuse to answer         right now, no guarantee
     until I can confirm."      it's current."
    --> unavailable             --> possibly wrong
A

Then which one do you choose, for this platform?

B

It's not one global answer — it depends on what's being read. A LAMF loan disbursement status must refuse to answer stale during a partition; showing "not yet disbursed" when it actually was risks a duplicate transfer. A customer's read-only "your last 10 transactions" view can happily serve a few-seconds-stale answer — nothing breaks if it's slightly behind.

A

So the trade-off isn't decided once, at the architecture level — it's decided per operation?

B

Exactly — and that's the honest answer, not a dodge. Pretending one global CP-or-AP choice fits every read in the system is the buzzword version of this problem. The real design has to let each operation state its own requirement.

Where this leaves us: consistency versus availability isn't a setting you flip once at system design time — it's a decision the system has to be capable of making differently for every single read. That capability doesn't exist yet; building it is exactly what the next two stages are for.
Engineering Philosophy

Consistency vs availability is not a one-time system-level choice. Different data needs different answers. A user's current balance needs consistency. A product recommendation feed can tolerate staleness. Design each read's guarantee separately.

Stage 6

What "eventual" actually has to mean

Architect A

You've said "eventually consistent" twice now like it's self-explanatory. Eventually when, and consistent according to whom?

✎ think first

If two replicas hold different values for the same key because one missed an update, what specific piece of information does the system need to know which value is actually newer?

Reveal the dialoguereveal
B

"Eventually" means: once a replica has received every update the leader has sent, it converges to the same value. It's not a vague promise — it's a guarantee that stops mattering the instant the replica catches up.

A

How does a replica know it's behind, and by how much?

B

Every write carries a version — a logical timestamp or sequence number from the leader. A replica compares the version it's holding against the version it just received; if the incoming one is newer, it overwrites. That's the entire mechanism behind "eventually."

LEADER WRITE LOG:
  v1: balance=1000
  v2: balance=1500   <- follower missed this one (network blip)
  v3: balance=1200

FOLLOWER catches up:
  has v1 --> receives v2 --> receives v3 --> now matches leader
  "eventually" = the moment it has processed every version in order
A

A client reads from a stale follower mid-catch-up and sees v1, then immediately reads again and sees v3. Is that a bug?

B

It's a real, named phenomenon — read-your-own-writes and monotonic-read violations — and no, it isn't automatically a bug, it's an explicit property this design is choosing to allow in exchange for availability. Whether it's acceptable depends entirely on what's being read, which is exactly what stage 5 already established.

Where this leaves us: "eventual consistency" stops being a hand-wave once it's backed by versions every replica can compare. But versioning alone only tells you which value is newer when there's one linear order of writes — it says nothing about what happens when two writes happen at the same time, from two different places. That's next.
Stage 7

The hardest question: two writes, at the same time, no leader to arbitrate

Architect A

A mobile app caches a customer's LAMF loan request offline. Two devices, briefly disconnected from each other, both submit an update to the same field at nearly the same instant. There is no single leader that saw both. Who wins?

✎ think first

If you can't say which write happened "first" — because there's no shared clock precise enough to prove it — what's the only honest thing the system can say about the two writes?

Reveal the dialoguereveal
B

The honest answer is: neither "won." They're concurrent — not because they happened at literally the same millisecond, but because neither write's context included knowledge of the other. Picking a winner by wall-clock time is picking based on a clock that was never reliable enough to trust.

LAST-WRITE-WINS (by timestamp):
  Device A: 14:02:00.001 --> amount=5000
  Device B: 14:02:00.002 --> amount=7000
  "B is later, B wins" -- but clocks drift, NTP isn't perfect,
  and neither device knew the other one existed.
A

So last-write-wins is dishonest. What's the alternative that isn't just a fancier guess?

B

Track causality, not time. Each write carries a version vector — which versions of the record it was built from. If neither write's vector contains the other's version, they're provably concurrent, not just probably close in time — and the system says so explicitly instead of silently picking one.

VERSION VECTORS:
  Device A writes based on version [x:1] --> produces [x:2, a:1]
  Device B writes based on version [x:1] --> produces [x:2, b:1]
  Neither vector contains the other --> PROVABLY concurrent
  --> surface BOTH to the application. Don't guess. Ask.
A

Now the application has to resolve a merge conflict. Isn't that just moving the hard problem somewhere else?

B

It's moving the problem to the only place that actually has the context to resolve it correctly — a loan amount conflict needs business logic, maybe a human, to decide. A storage engine silently guessing based on an untrustworthy clock isn't solving the conflict; it's hiding it.

Where this leaves us: the store's honest job is to detect and preserve conflicts, not to invent a fake resolution. That single decision — surface concurrency instead of erasing it — is what makes this a trustworthy system for financial data instead of a fast one that occasionally lies.
Stage 8

What actually writes to disk, and why it isn't a B-tree

Architect A

Every node still needs to persist data to disk somehow. A B-tree is the default choice in every relational database you've used. Why would you use anything else?

✎ think first

A B-tree updates a value by finding its exact page on disk and rewriting it in place. What does "finding the exact page" cost when the write rate is the thing you're optimizing for?

Reveal the dialoguereveal
B

A B-tree write means: seek to the right page, possibly split it, rewrite it in place. Every write is a small random write somewhere on disk — and random writes are the exact opposite of what we optimized for in stage 1.

B-TREE WRITE:
  find the page for this key -> maybe 3-4 disk seeks
  rewrite that page in place -> RANDOM write location

We chose write-throughput as our one goal in stage 1.
Random writes are the slowest thing a disk does.
A

So what's the alternative — never overwrite anything?

B

An LSM-tree: every write is appended sequentially to an in-memory buffer, which is periodically flushed to disk as a new, immutable sorted file. No seeking, no in-place rewrite — just append, exactly like the sequential-write argument that made sense for the write path from the start.

LSM-TREE WRITE:
  new write --> append to in-memory table (memtable)
  memtable fills --> flush AS-IS to a new sorted file on disk (SSTable)
  never rewritten in place -- old files just accumulate
A

"Old files just accumulate" — say that back to me and tell me why that's not obviously a disaster.

B

Because it is a disaster, deferred — a read now has to check the memtable, then potentially every accumulated file on disk, newest to oldest, to find the current value of one key. Fast writes were bought by making reads progressively worse. That bill comes due next.

Where this leaves us: the LSM-tree isn't a free upgrade over the B-tree — it's a direct trade of read cost for write cost, chosen because we declared write throughput the priority back in stage 1. Stage 9 is where that deferred read cost has to actually get paid down.
Stage 9

Paying down the bill: compaction

Architect A

A year of writes to one hot key means dozens of SSTable files, each possibly holding an old version of it. A read for that key now checks every one of them. How is that not a slow-motion collapse?

✎ think first

Most of those old files contain values that have since been overwritten. What's the minimum work needed to throw that dead weight away without ever blocking a live write?

Reveal the dialoguereveal
B

It would collapse, left unmanaged. Compaction runs as a background process: it merges several SSTables into one, keeping only the newest version of each key and discarding everything superseded. It never touches the live write path.

BEFORE COMPACTION:                 AFTER COMPACTION:
  File1: key=X, v1                   File_merged: key=X, v5 (only newest)
  File2: key=X, v3                   -- v1, v2, v3, v4 discarded,
  File3: key=X, v5                      they were dead weight
  Read for X checks all 3 files      Read for X checks 1 file
A

Compaction reads and rewrites huge amounts of data continuously. Doesn't that just relocate the disk I/O problem you claimed to have solved?

B

It relocates it deliberately, to a background process running on the disk's spare I/O budget, instead of on the critical path of a user's write. The user-facing write is still one sequential append; compaction's cost is paid asynchronously, amortized, not synchronously, per request.

A

What happens if compaction falls permanently behind write volume?

B

Then read amplification climbs without bound and the system degrades exactly the way you're implying — which is precisely why compaction throughput and I/O priority have to be explicitly budgeted and monitored, not left to run "whenever there's spare capacity" and hoped about.

Where this leaves us: the LSM-tree's write speed was never free — it was a loan against future read performance, and compaction is the repayment schedule. A store that appends fast but never compacts isn't fast, it's just delaying the collapse Architect A already predicted.
Stage 10

Finding a record without knowing its key

Architect A

Everything so far assumes you know the exact key — the customer ID. Now find every insurance policy where the premium is overdue. There's no key for "overdue." What do you do?

✎ think first

A secondary index has to live somewhere. If it lives on the same node as the primary data it indexes, what happens the instant that data is repartitioned across nodes by a different key?

Reveal the dialoguereveal
B

Two options, and they fail in opposite directions. A local index — each node indexes only the records it holds — is cheap to update but a query still has to scatter-gather across every node, the exact problem stage 2 already ruled out.

LOCAL INDEX (per node):
  Node1's index: overdue policies ON Node1
  Node2's index: overdue policies ON Node2
  Query "all overdue" --> ask every node again -- scatter-gather returns
A

So build one global index instead.

B

A global index — partitioned by the indexed field itself, "overdue" living on its own node regardless of where the underlying policy lives — answers the query in one lookup. But now a single write to one policy has to update two different partitions on two different nodes: the primary record, and the index entry.

GLOBAL INDEX:
  Write policy P (lives on Node3, keyed by customer)
    --> ALSO update overdue-index (lives on Node7, keyed by status)
  ONE logical write is now TWO physical writes, on TWO different nodes.
A

What happens if the second write — the index update — fails after the first one succeeds?

B

The index goes stale relative to the data it describes, and it has to be treated as eventually consistent — with its own catch-up mechanism, the same version-based repair from stage 6, applied a second time to a second kind of data. There's no version of a global secondary index that's both fast to write and instantly consistent; you're choosing between fast local writes with scattered reads, or fast global reads with a second write that can lag.

Where this leaves us: a secondary index isn't a free lookup feature bolted onto a key-value store — it's a second write path with its own consistency problem, inheriting every trade-off already fought for in stages 5 through 7, now duplicated.
Stage 11

Hashing was supposed to fix this

Architect A

Hashing spread customers evenly across nodes back in stage 3. Now one large AMC's mutual fund scheme generates ten thousand times the order volume of a typical retail customer, and it hashes onto one node. Explain why "even distribution" didn't save you.

✎ think first

Hashing guarantees an even spread of keys across nodes. Does it guarantee an even spread of the work each key generates?

Reveal the dialoguereveal
B

No — and that's the flaw in how I described hashing back in stage 3. It spreads keys evenly. It says nothing about the volume of writes each key attracts. One scheme's key can carry ten thousand times the write load of another and hashing has no way to know that.

HASH DISTRIBUTES KEYS EVENLY:            BUT NOT WORKLOAD:
  Node1: scheme_A, scheme_D                scheme_A: 2 orders/sec
  Node2: scheme_B, scheme_E                scheme_D: 20,000 orders/sec
  Node3: scheme_C, scheme_F                Node1 is silently on fire.
A

So detect the hot key and move it to its own node?

B

Reactive rebalancing helps, but it's always chasing a problem that's already hurting someone. The proactive fix is salting — append a small random suffix to the hot key itself, splitting its writes across several sub-keys on several nodes, then merge results back together at read time.

WITHOUT SALT:            WITH SALT (N=4):
  scheme_D -> Node1        scheme_D#0 -> Node1
  ALL writes, one node     scheme_D#1 -> Node2
                           scheme_D#2 -> Node3
                           scheme_D#3 -> Node4
                           writes for scheme_D now spread across 4 nodes
A

Doesn't that make every read for that key four times more expensive, forever, even after the spike passes?

B

Yes — which is exactly why salting should be applied selectively, to identified hot keys only, not uniformly to every key in the store. It's a targeted trade: pay a small, permanent read tax on a few known-hot keys to avoid an unbounded write bottleneck on them.

Where this leaves us: uniform key distribution was never the same guarantee as uniform load distribution — real-world data isn't uniform, and a design that assumes it is will always eventually meet its one outlier key that breaks the assumption.
Stage 12

Making the CAP trade a dial, not a switch

Architect A

Stage 5 said consistency-versus-availability has to be decided per operation. You haven't actually built that yet — you've only argued it should exist. Build it.

✎ think first

With 3 replicas per partition, how many of them need to agree on a write, and how many need to agree on a read, so that a write and a read can never both succeed on entirely disjoint sets of stale replicas?

Reveal the dialoguereveal
B

Quorums. With N replicas, require W replicas to acknowledge a write and R replicas to respond to a read, such that W + R > N. That inequality guarantees any read set and any write set must share at least one replica — the one that's guaranteed to have the latest value.

N=3 replicas.  W=2, R=2.  W+R=4 > N=3.

Write set {Replica1, Replica2} acknowledges.
Read set  {Replica2, Replica3} is queried.
Replica2 is in BOTH sets --> read is guaranteed to see the write.
A

So this is just consistency again, dressed up in different notation?

B

No — it's a dial, not the old binary switch. A LAMF disbursement write can demand W=3, R=1 — strict on write, tolerant on read. A read-heavy analytics dashboard can run W=1, R=1 — fast on both, weakly consistent, and that's an acceptable, explicit choice for that specific operation.

PER-OPERATION TUNING:
  critical write:   W=3 (all must ack)  -- slow, safe
  casual read:      R=1 (any one answers) -- fast, may be stale
  Same store. Different guarantees. Chosen by the caller, per call.
A

What's the cost of setting W and R too low across the board, out of laziness?

B

You silently reintroduce every problem stages 5 through 7 already solved — stale reads that look fresh, conflicting writes that never surface. The quorum inequality is a guarantee, not a suggestion; violating it isn't a performance trade, it's opting back out of correctness without saying so.

Where this leaves us: the abstract "it depends per operation" argument from stage 5 now has a concrete knob — W and R — that any caller can set explicitly. The CAP trade-off didn't go away; it became a parameter instead of an architectural constant.
Stage 13

The replica that missed a write and never asked

Architect A

A replica was down for four hours — a hardware blip, nothing dramatic. It's back online now. Nobody told it what it missed. How does it find out, without you manually intervening at 3 a.m.?

✎ think first

Comparing every single record between two replicas byte-by-byte to find the differences after an outage — what's wrong with that as a strategy, at scale?

Reveal the dialoguereveal
B

Two complementary mechanisms, not one. First, hinted handoff — while the replica was down, a healthy node temporarily held the writes meant for it, as a "hint," and replays them the moment it comes back. That covers short outages cheaply.

HINTED HANDOFF:
  Replica3 is DOWN --> Replica1 stores a HINT: "this write was for Replica3"
  Replica3 comes back --> Replica1 replays its stored hints to Replica3
  -- fast catch-up for SHORT outages
A

Four hours is longer than most hint-retention windows. What covers the outage hints didn't survive?

B

Anti-entropy, running as a permanent background process, using Merkle trees to compare replicas efficiently. Instead of comparing every record, each replica builds a tree of hashes — compare the top hash first; if it matches, the entire subtree matches and you stop there. Only mismatched branches get compared further down.

MERKLE TREE COMPARISON:
  Root hash matches?  YES --> done, entire replica set matches, O(1) check
  Root hash differs?  --> compare children --> only recurse into
                            the branches that actually disagree
A

Why not just always run a full comparison — isn't the tree an unnecessary complication?

B

Because a full comparison costs O(all records) every single time you run it, even when nothing's wrong — which is the common case. The tree turns "verify billions of records agree" into "verify one hash agrees," and only pays the expensive cost on the actual mismatched fraction.

Where this leaves us: hinted handoff and anti-entropy aren't redundant — one is a fast reflex for short gaps, the other is a slow, thorough safety net for anything the reflex missed. Together they're what lets "eventually consistent," first promised back in stage 6, actually keep its promise without a human watching every replica by hand.
