FOUNDATION

First rough sketch — what are we really building?

▶

A NoSQL database looks simple from outside:

App --> put/get data

But the real questions are:

  • How is data partitioned?
  • How is it replicated?
  • How do reads find the right node?
  • How do writes stay durable?
  • How do we balance consistency vs availability?
  • How do indexes work at scale?
  • How do we recover from failures?

That is the real system design.

Summary

partitioned storage
+ replicated state
+ distributed read/write coordination
+ consistency tradeoffs
+ storage-engine design

Core job: store huge data across many machines while still answering reads/writes reliably.
Engineering Philosophy

Before you pick a database type, write down your three most important read patterns. The database shape should follow the query shape, not the other way around.

PROBLEM 01

What kind of NoSQL database are we designing?

▶

Before design, first define the model. Because “NoSQL” is not one thing.

Main families

1. Key-Value
   key -> value

2. Document
   key -> semi-structured document

3. Wide-Column
   partition key + clustering/order

4. Graph
   nodes + edges + traversals

Rough sketch

Key-value
user:1001 -> {blob}
cart:77   -> {blob}
Document
user:1001 -> {
  name: "Ankur",
  city: "Chennai",
  orders: [...]
}
Wide-column
Partition: user=1001
Rows:
2026-04-01 -> login
2026-04-02 -> payment
2026-04-03 -> logout
PROBLEM 02

The real heart is partitioning

▶

The first deep distributed problem is: where does a record live?

Because one machine cannot hold everything. So we shard/partition data.

key = user:1001
      |
      v
 hash(key)
      |
      v
 partition 17
      |
      v
 node N2

Partitioning view

All data
  |
  +--> Partition P1 -> Node A
  +--> Partition P2 -> Node B
  +--> Partition P3 -> Node C

Two common strategies

1. Range partitioning
user_id 1-10000     -> shard 1
user_id 10001-20000 -> shard 2
Good

range scans easier

Bad

hotspots possible

2. Hash partitioning
hash(user_id) % N
Good

better spread

Bad

range queries harder

Engineering Philosophy

The partition key is the most important design decision in a distributed database. A bad key creates hot spots that no amount of hardware can fix.

PROBLEM 03

Replication is non-negotiable

▶

Partitioning alone is not enough. If one node dies, data should not vanish. So each partition is replicated.

Partition P17

Primary / Leader -> Node A
Replica          -> Node B
Replica          -> Node C

Or leaderless/quorum style:

Partition P17 copies on A, B, C
all can participate in quorum logic
Strong mental model — partitioning decides where data is split. Replication decides how each split survives failure.

PROBLEM 06

Consistency model is one of the defining differences

▶

This is where NoSQL systems differ a lot. Possible models:

strong consistency
eventual consistency
read-your-writes
quorum consistency
leader-based consistency
causal-ish/session guarantees

Strong consistency sketch

Write acknowledged
then future reads must see it

Eventual consistency sketch

Write acknowledged
some replicas may still be old briefly
reads may differ temporarily
eventually converge

Quorum sketch

Replication factor = 3

Write quorum W = 2
Read quorum  R = 2

If R + W > 3,
freshness can be stronger
Engineering Philosophy

Consistency is not free. Every write that needs a strong guarantee pays a coordination cost. Decide how much consistency you actually need before you pay for it.

PROBLEM 08

Indexing is much harder than it looks

▶

Primary key lookup is easiest. Secondary indexes are harder because data is partitioned elsewhere.

Main record:
user_id -> document

Need query:
find by email

Now where does email index live?

email -> user_id

This itself becomes distributed.

Index write path

write document
   |
   +--> update primary storage
   +--> update secondary index
Now consistency issue appears.
PROBLEM 09

Storage engine choice changes the whole design

▶

A NoSQL DB often uses one of two internal styles.

A. B-tree style thinking

  • data updated in place-ish / page-oriented
  • good for point lookups and range access

B. LSM / log-structured style thinking

  • writes append fast
  • flush immutable files
  • compact later

This is very common in modern NoSQL stores.

LSM rough sketch

write
  |
  v
WAL
  |
  v
memtable
  |
 flush
  v
SSTable files
  |
 compaction
  v
merged files
Why LSM is popular: fast writes, good for write-heavy distributed systems. But read path and compaction become more complex.
Engineering Philosophy

The storage engine isn't just an implementation detail — it sets your write/read trade-offs for the life of the system. Choose it as deliberately as you choose your data model.

PROBLEM 11

Rebalancing is a major distributed systems problem

▶

As data grows, nodes are added/removed. Then partitions may move.

Before:
P1 -> Node A
P2 -> Node B
P3 -> Node C

Add Node D

After:
some partitions/splits move to D

Why this is hard

  • moving data takes time
  • traffic continues
  • writes may still arrive
  • ownership must switch safely
PROBLEM 12

Failure handling is the real test

▶

Failures define whether the design is mature.

Failure map

Client -> Router -> Leader -> Replicas -> Disk

Failure can happen anywhere.

Key cases

Case 1 — Node crash
leader dies
new leader must take over

Case 2 — Network partition
some replicas can't talk
who is authoritative?

Case 3 — Disk crash / corruption
need log replay / replica recovery
PROBLEM 13

Repair and anti-entropy matter

▶

Replicas can drift. Need repair mechanisms.

Replica A has latest key version
Replica B missed some update
Replica C is old
Repair process:
compare state / logs / Merkle-like summaries
sync missing data
PROBLEM 14

Transactions in NoSQL are different

▶

Many NoSQL systems started with:

single-key / single-document atomicity

Multi-key distributed transactions are much harder.

single document update  -> easy atomic unit

multi-partition update  -> much harder
PROBLEM 15

Query model defines practical usefulness

▶

NoSQL power is often not rich SQL joins. So query design must be explicit.

Good fit

point lookup
partition-local scans
time-ordered reads
document fetches

Harder fit

ad hoc joins
global aggregations
cross-partition sort-heavy queries

PROBLEM 16

TTL, compaction, and archival are often first-class

▶

Many NoSQL use cases are logs, sessions, events, time-series, cache-like data. So these become very important:

  • TTL
  • expiration
  • tombstones
  • compaction
  • cold archival
session:123
created at 10:00
TTL = 24h

After expiry:
data may become logically deleted
physical cleanup may happen later
PROBLEM 17

Multi-region adds another level of difficulty

▶

Now replication is not only within one data center.

Region A <------> Region B <------> Region C

Choices:

  • single-writer multi-region replicas
  • multi-writer replication
  • leader per region
  • global tables style

take data
