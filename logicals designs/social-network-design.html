PROBLEM 01

How should we model follow / unfollow?

▶

A naive design says:

Table: follows
follower_id
followee_id

Is this only “follow”? Or is this a generic relationship system?

Better abstraction

Table: relations

from_user_id
to_user_id
relation_type
state
position
created_at
updated_at

Example:

from=A
to=B
relation_type=FOLLOW
state=ACTIVE
position=timestamp
+----+         +----+
| A  | ------> | B  |
+----+         +----+

Stored as edge:
(from=A, to=B, type=FOLLOW, state=ACTIVE)

Why “relations” is a stronger design than “follows”

Because the product grows.

Today:
    follow

Tomorrow:
    close friend
    mute
    block
    subscribe
    favorite creator
    hidden connection
    private circle
So low-level naming should be one level more abstract. Weak abstraction: follows. Stronger abstraction: relations.
Engineering Philosophy

A social graph is read far more than it is written. Design the storage for how data is read, not how it was created.

PROBLEM 02

Why do we often store two edges?

▶

When A follows B, product asks two kinds of questions:

- whom do I follow?
- who follows me?

If we only store one row:

   A -> B

Then one direction is easy, the reverse may become expensive depending on sharding/indexing. So many systems store two directional entries.

A follows B

and

B followed_by A
Single relationship in product: A follows B

Stored internally as 2 edges:

1) A --FOLLOWING--> B
2) B --FOLLOWED_BY--> A

Or visually:

         +----+                +----+
          | A  |                | B  |
          +----+                +----+
             \                    /
              \                  /
      FOLLOWING \              / FOLLOWED_BY
                 \            /
                  \          /
                   +--------+

More practical sketch:

=======================
Row 1: source=A, dest=B, type=FOLLOWING
Row 2: source=B, dest=A, type=FOLLOWED_BY

Why this helps

Now these queries become local and cheap:

who do I follow?      -> source = A, type = FOLLOWING
who follows me?       -> source = A, type = FOLLOWED_BY
PROBLEM 03

How should the primary key / index be shaped?

▶

A beginner thinks:

PRIMARY KEY (source_id, destination_id)

But an architect asks: What are my dominant query patterns?

If main query is:

  • show followers of X
  • newest first
  • paginate fast
  • skip deleted
  • fetch next page by cursor

then the query pattern matters more than “uniqueness elegance”.

Possible model

relations
---------
source_id
destination_id
relation_type
state
position

where:

  • position could be timestamp / snowflake / score
  • state could be ACTIVE / DELETED / ARCHIVED
INDEX (source_id, relation_type, state, position DESC, destination_id)

or if only one relation type exists for this table:

INDEX (source_id, state, position DESC, destination_id)

And key/index can be shaped around read pattern.

For source = A

+---------------------------------------------+
| source | state  | position | destination    |
+---------------------------------------------+
| A      | ACTIVE | 9005     | U17            |
| A      | ACTIVE | 9004     | U88            |
| A      | ACTIVE | 9003     | U02            |
| A      | DELETD | 9002     | U13            |
+---------------------------------------------+

Why this index

WHERE source_id = A
 AND state = 'ACTIVE'
ORDER BY position DESC
LIMIT 20

So index should start with:

==========================
source_id       -> find all edges of A
state (+ type)  -> skip deleted/inactive rows early
position DESC   -> already sorted newest first
destination_id  -> last column / covering / uniqueness support
PROBLEM 04

Why cursor pagination matters so much

▶

Offset pagination becomes expensive at deep pages.

Bad:
   SELECT ...
   FROM relations
   WHERE source_id = A
   ORDER BY position DESC
   LIMIT 20 OFFSET 100000;
Better:
   WHERE source_id = A
   AND position < last_seen_position
   ORDER BY position DESC
   LIMIT 20;
Bad:
[read first 100000 rows] -> discard -> return 20
Good:
jump directly to cursor position -> return 20
Page 1:
9005, 9004, 9003

Client sends last_seen = 9003

Page 2:
position < 9003
=> 9002, 9001, 9000
PROBLEM 05

How should we shard the relation graph?

▶

A common approach: shard by source_user_id.

Why? Because many core queries are:

show people I follow
show my followers
paginate my adjacency list
shard = hash(source_user_id) % N

source A -> shard 2
source B -> shard 7
source C -> shard 1

All adjacency for source A stays together in one shard. That avoids cross-shard query for source-based scans.

But now the hard question

If celebrity X has 200 million followers:

- source = X
- huge adjacency list
- single logical hotspot
Now shard-by-source creates giant hot partitions.
Engineering Philosophy

Celebrities create hot partitions. Shard a social graph for your worst-case user — the one with 10 million followers — not for the average one.

PROBLEM 06

What kind of graph is this really?

▶

People hear “social graph” and jump to graph database. But not all graph problems are the same. There are roughly two worlds:

World A — shallow adjacency graph

Queries like:

  • who follows me?
  • who do I follow?
  • mutuals
  • followers count

World B — deep traversal graph

Queries like:

  • 2-hop / 3-hop paths
  • people you may know via multiple relationship hops
  • graph shortest path / community search

Twitter-like follow graph mostly leans toward large adjacency lists, shallow traversals. LinkedIn-style “2nd degree”, “3rd degree”, professional path graph leans more into traversal richness.

Twitter-like:
A -> [very large list of edges]
Need fast read/write on huge adjacency

LinkedIn-like:
A -> B -> C -> D
Need traversal intelligence
Query - go through my friend list
        -> go to all who I follow
PROBLEM 07

Designing the feed is the true battlefield

▶

A social network without feed is just storage. The real magic is:

Who posted?
Who should see it?
When should they see it?
In what order?

Feed is where scale and product meet.

1. Fanout-on-write

When the author posts, push feed entry to followers.

Author posts
   |
   v
find followers
   |
   v
write feed entry into follower timelines

2. Fanout-on-read

When the reader opens the app, fetch posts from people they follow and assemble live.

Reader opens feed
   |
   v
find followees
   |
   v
fetch recent posts from them
   |
   v
merge + sort + rank

Hybrid feed design — the practical answer

A strong real-world idea:

  • normal users → fanout on write
  • celebrities / huge fanout accounts → fanout on read
               +----------------------+
               | New Post by Author   |
               +----------+-----------+
                          |
               +----------+-----------+
               | fanout size check    |
               +----------+-----------+
                          |
              +-----------+------------+
              |                        |
              v                        v
      normal author              celebrity author
      fanout on write            fanout on read
Engineering Philosophy

Push-on-write means fast reads but expensive writes for large followings. Pull-on-read is cheap to write but slow to serve. Most systems end up with a hybrid based on follower count.

PROBLEM 08

What exactly is a feed item?

▶

A feed item is not necessarily the full post.

feed_item
---------
viewer_id    -> who will see the post
post_id      -> which post to show
producer_id  -> who created the post
reason       -> why this post is shown
score        -> how important/relevant this post is
created_at   -> when this feed entry was created

Then the full post can be fetched separately. This makes fanout lighter.

Feed table for U1

+-----------------------------------------+
| viewer | post_id | producer | score     |
+-----------------------------------------+
| U1     | P90     | A        | 0.92      |
| U1     | P81     | B        | 0.88      |
| U1     | P33     | C        | 0.71      |
+-----------------------------------------+
RecapSay it in your own words first — then open.＋

We are trying to solve how to store a user's home feed efficiently.

The key idea is that a feed item is usually not the full post itself. Instead, it is a light entry that says something like: “show post P90 to user U1, made by A, because of this reason, with this score.” Then the real post content can be fetched separately when needed.

This makes the feed table smaller, faster, and easier to fan out to many users. In short, we are not copying the whole post again and again into every user's feed — we are mostly storing a reference plus some feed-related metadata.

PROBLEM 09

Ranking is not just sorting by time

▶

Simple feed:

ORDER BY created_at DESC

Real social feed:

score =
    freshness
  + affinity
  + content quality
  + media boost
  + interaction velocity
  + creator importance
  - spam signals
Post candidates for user U

P1: new but weak affinity
P2: older but strong affinity
P3: viral content from close friend

score decides order
not time alone
RecapSay it in your own words first — then open.＋

We are trying to solve how posts should be ordered in the feed. A beginner may think feed should just show the newest post first, but real social apps do more than that.

Here we rank posts using many signals together: how new the post is, how close the viewer is to that creator, how engaging or high-quality the content is, whether it is getting fast reactions, and whether it looks spammy. So instead of saying “latest always wins,” the system says “best post for this user at this moment should come first.”

In short, ranking means the feed is ordered by importance and relevance, not only by time.

Engineering Philosophy

Ranking is a product decision wrapped in an engineering problem. "Most relevant" is not a database query — it requires signals, freshness rules, and a computation budget someone has to pay for.

PROBLEM 10

How should reactions be modeled?

▶

Do not create a narrow likes table if reaction types can expand.

Better abstraction

reactions
---------
user_id
post_id
reaction_type
state
created_at

Possible values:

LIKE
LOVE
HAHA
WOW
SAD
ANGRY

And usually one user should have only one active reaction per post.

User U reacts to Post P

(U, P, LIKE)

Later changes to LOVE

(U, P, LOVE)

Need uniqueness by:

UNIQUE(user_id, post_id)

if only one active reaction allowed.

RecapSay it in your own words first — then open.＋

We are trying to solve how to store reactions on a post in a smart way. Instead of making a small table only for “likes,” the idea is to make a more general reactions table, because today it may be like, but tomorrow it can also be love, haha, wow, sad, or angry. So we store the user's reaction in one flexible structure with fields like user, post, reaction type, and state.

If one user is allowed to have only one active reaction on one post, then we keep a uniqueness rule on (user_id, post_id). That means the same user cannot keep both LIKE and LOVE active on the same post at the same time — if they change reaction, we update that one row instead of creating many rows.

PROBLEM 11

Counting is deceptively hard

▶

Follower count, following count, like count, comment count, unread count. These look simple, but become hot.

Bad idea:
SELECT COUNT(*)
FROM relations
WHERE source = X AND type = FOLLOWED_BY AND state = ACTIVE

on every profile render.

Better

  • maintain materialized counters
  • update asynchronously
  • allow slight lag
User follows A
   |
   v
write relation edge
   |
   v
emit event
   |
   v
counter updater
   |
   v
A.follower_count += 1
RecapSay it in your own words first — then open.＋

We are trying to solve how to show numbers like follower count, like count, comment count, or unread count without making the system slow. At first it looks easy — just count rows from the database each time. But on a big social app, doing that on every profile or post view becomes expensive.

So we keep ready-made counters that get updated when actions happen, often through an async background flow. In short, instead of counting everything again and again, we store the number separately and update it as new actions come in. This makes the app fast, even if the count is slightly delayed for a few seconds.

Notifications are not same as feed

▶
Very important distinction.

A user may receive:

- a notification
- a feed item
- both
- neither

Example:

  • someone liked your post → notification maybe yes
  • celebrity posted → feed yes, notification maybe no
  • DM received → notification yes, feed no
Action happens
   |
   +--> feed candidate
   |
   +--> notification candidate
   |
   +--> unread badge update

These are different pipelines.

A notification is a direct alert to a user, like “someone liked your post” or “you got a message.”

PROBLEM 13

Messaging unread indicator

▶

A near-real-time unread indicator is not only “count”. Sometimes product wants:

  • count of unread threads
  • latest senders
  • instant websocket badge
  • reset when user opens inbox

So an in-memory fast store is useful.

A sends msg to B
      |
      v
 save message durable
      |
      v
 update unread state in Redis
      |
      v
 if B online -> push badge via WebSocket

Sorted set style idea

Instead of only count:

Unread senders for B = [i, j, k, b]

Then:

count = size of set
UI can show names
clear on inbox-open
RecapSay it in your own words first — then open.＋

We are solving one simple product problem: how to show an unread message badge fast, like “3 unread” on chat or inbox.

But unread is not always just a number — sometimes product also wants to show which senders are unread, update the badge instantly, and clear it as soon as the user opens messages. So the idea is: first save the actual message safely in the database, then update a very fast store like Redis for unread info, and if the user is online, push the new badge instantly through WebSocket.

In short, the DB is used to safely keep the real message, while Redis is used to quickly show and update the unread badge in real time.

PROBLEM 15

Tags on photos / relative coordinates

▶

Absolute pixels are fragile because image dimensions vary after resize.

Better: normalized coordinate ratios, or bounding box ratios.

Image width  = W
Image height = H

Tag box:
x1/W, y1/H, x2/W, y2/H
Instead of:
(400px, 210px)
Store:
(0.52, 0.31)

So same tag still works after image resize.

RecapSay it in your own words first — then open.＋

Facebook does not store photo tags using fixed pixels like 400px from the left and 210px from the top, because that breaks when the same photo is resized into a thumbnail, preview, or full-screen version.

Instead, it stores the tag position as a ratio or percentage of the image size, such as 0.52 of the width and 0.31 of the height. This means the tag stays on the same face even when the photo becomes bigger or smaller. In simple words, instead of saving the exact distance, it saves the relative position inside the photo.

PROBLEM 17

Blocking, muting, privacy

▶

This changes many downstream systems.

If A blocks B:

  • B should maybe not see A profile
  • B should maybe not send messages
  • A's content should disappear from B feed
  • recommendation should stop suggesting them
So block is not just one edge — it changes many read paths.
A blocks B
   |
   +--> relation updated
   +--> feed visibility changes
   +--> messaging permissions change
   +--> search visibility changes
   +--> recommendation suppression
