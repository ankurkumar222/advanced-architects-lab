FOUNDATION

Normal scheduler vs distributed scheduler

▶
Normal scheduler:
one machine
   |
   +--> runs jobs
Distributed scheduler:
many machines
   |
   +--> share jobs
   +--> balance load
   +--> survive machine failure
FOUNDATION

What is the real heart of this system?

▶

Not API. Not UI. Not dashboards. The real heart is this:

(1) Know WHICH job is due
(2) Ensure ONLY ONE scheduler claims it
(3) Ensure SOME worker executes it
(4) Recover if something dies in between

Main problem is NOT:

"how to run background jobs?"

Main problem IS:

"how to run the right job,
at the right time,
on one owner,
with safe recovery,
under failure,
without dangerous duplicates?"
Engineering Philosophy

The heart of a scheduler isn't timing — it's making sure the same job doesn't run twice. Design the ownership guarantee first, then worry about when to run.

FOUNDATION

Job definition vs job execution

▶
JOB DEFINITION  !=  JOB EXECUTION
Job Definition
--------------
"Run report every hour"

Execution records
-----------------
10:00 run -> success
11:00 run -> fail
11:05 retry -> success
12:00 run -> success

If you mix both into one row, complexity rises fast. Think of it like this: Job Definition is the rule, and Job Execution is each actual attempt to follow that rule.

Example: “Run report every hour” is just one definition. But every time it actually runs — 10:00 success, 11:00 fail, 11:05 retry success, 12:00 success — those are separate execution records.

If you keep both in one row, that one row has to keep changing for every attempt, retry, failure, and history entry, which becomes messy. So we separate them: one row says what should happen, many rows say what actually happened each time.

JOB_DEFINITION
--------------
job_id = 101
name = hourly_report
schedule = every 1 hour
next_run_time = 12:00
status = ACTIVE
payload = report_config_A
JOB_EXECUTION
-------------
exec_id = 9001   job_id = 101   slot=10:00   attempt=1   SUCCESS
exec_id = 9002   job_id = 101   slot=11:00   attempt=1   FAILED
exec_id = 9003   job_id = 101   slot=11:00   attempt=2   SUCCESS
exec_id = 9004   job_id = 101   slot=12:00   attempt=1   RUNNING

How scheduler should work

Step 1  Scheduler reads only JOB_DEFINITION
        Find jobs where next_run_time <= now

Step 2  Create a new JOB_EXECUTION row for that run
        For 11:00 slot: create execution record

Step 3  Send that execution to worker

Step 4  Worker updates only JOB_EXECUTION
        not the definition again and again

Step 5  After dispatch, update JOB_DEFINITION.next_run_time
        to next schedule time
JOB_DEFINITION
   |
   | scheduler sees next_run_time = due
   v
create JOB_EXECUTION row
   |
   v
dispatch to worker
   |
   v
worker runs
   |
   v
update JOB_EXECUTION status

Because now both things are clean:

Definition table answers

  • what job is this?
  • when should it run next?
  • is it active?

Execution table answers

  • what happened at 10:00?
  • did retry happen?
  • how many failures?
  • how long did it take?
If you mix both in one row, that becomes confusing.
PROBLEM 01

How do we know which jobs are due?

▶

Time line

09:00 ---- 09:01 ---- 09:02 ---- 09:03 ---- 09:04

Job A due at 09:01
Job B due at 09:03
Job C due at 09:03
Job D due at 09:04

Store next_run_time:

+--------------------------------------+
| job_id | next_run_time | status      |
+--------------------------------------+
| A      | 09:01         | scheduled   |
| B      | 09:03         | scheduled   |
| C      | 09:03         | scheduled   |
| D      | 09:04         | scheduled   |
+--------------------------------------+

Scheduler asks:

SELECT jobs
WHERE next_run_time <= now
AND status = 'scheduled'
Give me all jobs whose planned run time has come or passed, and which have not yet started / completed / failed.

PROBLEM 02

How do we stop two schedulers from picking the same job?

▶
           +-------------+
           | Job X due   |
           +-------------+
              /       \
             /         \
            v           v
       Scheduler S1   Scheduler S2

Both see same row. Both try to dispatch. That causes duplicates.

Option A — Claim with lease

Before claim:
+----------------------------------+
| Job X | due | unclaimed          |
+----------------------------------+

S1 does atomic claim:
    claimed_by = S1
    lease_until = now + 30s

+--------------------------------------------------+
| Job X | due | claimed_by=S1 | lease_until=09:05  |
+--------------------------------------------------+

Now S2 cannot claim it.

Option B — Shard ownership

hash(job_id) or hash(tenant_id)
          |
          v
      shard number
          |
          v
 owned by one scheduler node

Jobs
 |
 +--> shard 1 --> Scheduler S1
 +--> shard 2 --> Scheduler S2
 +--> shard 3 --> Scheduler S3

Then only the owner scans that shard.

Option A:
everyone can see job, but only one can claim it

Option B:
only one scheduler is allowed to look at that job's group
Thought-Q
Q6Which is better?＋
Lease per job
vs
Shard ownership + local claim
Q7If shard ownership is used, what happens when one scheduler dies?＋
S2 dies
who takes shard 2?
how fast?
how safely?
Q8If lease duration is too long, what happens?＋
worker/scheduler dies
job remains stuck until lease expires
Q9If lease duration is too short, what happens?＋
slow processing
lease expires too early
another node reclaims
duplicate dispatch
Q10Who is the source of truth for claim ownership?＋
  • DB row?
  • distributed lock service?
  • consensus layer?
  • partition ownership in queue?
Q11What is worse: temporary delay, duplicate run, or missed run?＋
Your answer changes the architecture.
Engineering Philosophy

A distributed lock can lie. If the machine holding the lock crashes before releasing it, the lock is held by a ghost. Always design for what happens when the lock holder disappears.

PROBLEM 03

Scheduler should decide time, but should it also execute?

▶
No.

Why not

Bad model:

Scheduler --> executes job directly

Problem: scheduler becomes —

  • time checker
  • business executor
  • retry engine
  • concurrency controller
  • failure handler

Too many roles.

Better

Scheduler ---> Queue ---> Worker

This split is very important.

Thought-Q
Q12What is the scheduler's real responsibility?＋

Pick one:

  • A) decide due jobs
  • B) execute jobs
  • C) both

Only one answer scales cleanly.

It should not. That is why dispatch and execution should be separated.

PROBLEM 05

Retry is not just retry

▶

Retry is actually business correctness design.

Attempt 1 ----fail----> wait 10s
Attempt 2 ----fail----> wait 1m
Attempt 3 ----fail----> wait 5m
Attempt 4 ----fail----> DLQ

But not every failure is same:

network timeout        -> retry maybe yes
5xx from downstream    -> retry yes
validation error       -> retry no
duplicate conflict     -> depends

Q23Should retries keep original schedule time or new execution time?＋
Open question — reason it out before moving on.
Q24Does retry preserve job ordering?＋
Job A fails
Job B succeeds

Now order changed

Is that acceptable?

Q25Where should retry state live?＋
  • queue only?
  • DB only?
  • both?
Q26Should failed jobs re-enter same queue or separate retry queue?＋
Open question — reason it out before moving on.
Engineering Philosophy

Retry and idempotency are different things. Retry is what the caller does. Idempotency is what the system guarantees. Build the system to be idempotent, and retries become harmless.

PROBLEM 06

Duplicate execution prevention

▶

In distributed systems, duplicates are normal unless you design carefully.

Duplicate paths

duplicate can happen because:

1) two schedulers dispatch same job
2) queue redelivers
3) worker lease expires too early
4) worker crashes after side effect
5) publish succeeded but ack lost

Important mental model

Exactly-once execution
is usually a dream at system boundary

What we usually do:
At-least-once processing
+
idempotent execution

Idempotency sketch

execution_key = job_id + scheduled_slot

same key comes again
downstream detects duplicate
and ignores second one

In a distributed system, "did this already run?" is not automatically a yes or no. Duplicate prevention is something you build deliberately — it doesn't come for free.

PROBLEM 09

Failure handling

▶

Failure point map

[DB stores job]
      |
      v
[Scheduler claims]
      |
      v
[Scheduler publishes]
      |
      v
[Worker receives]
      |
      v
[Worker executes]
      |
      v
[Result saved]

Failure can happen at every arrow.

Case 1 — Scheduler crashes after claim, before publish

claim success
publish not done
scheduler dies

Question: how is job recovered?

Case 2 — Publish succeeds, but scheduler crashes before marking dispatched

queue has message
DB may still show old state

Now what is truth?

Case 3 — Worker gets message, starts work, crashes mid-way

RUNNING
but unfinished

How to detect? How long to wait? Who retries?

Case 4 — Worker completes business side effect, but result update fails

side effect done
status not updated
Most dangerous practical case.
