STAGE MAP

Two stages

▶

For an SQS-like system, splitting into 2 stages makes it much easier to read.

This split is better because:

  • Stage 1 = how message enters safely
  • Stage 2 = how message leaves safely

Stage 1 — write path + storage path

  • Producer sends message
  • API routes it
  • partition leader stores and replicates it
  • message becomes available
Producer -> API -> Router -> Partition Leader -> Replicas -> Available Index

Stage 2 — read path + visibility timeout + delete/retry

  • Consumer receives message
  • queue marks it inflight
  • consumer processes
  • delete on success
  • redelivery on timeout/failure
Consumer -> Receive -> Inflight/Visibility Timeout -> Process -> Delete
                                         |
                                         v
                                   Timeout / Retry / DLQ

Before receive:
Message M1 = visible

A message is a unit of work, not just data. The envelope — ID, visibility timeout, delivery count — is what makes it possible to retry, deduplicate, and route failures safely.

PROBLEM 05

Visibility timeout is the real heart

▶

This is perhaps the most SQS-like part. A consumer should receive a message, but the queue should not delete it immediately.

Why? Because consumers may crash.

So queue does this: message delivered, but only hidden temporarily. This is a visibility timeout.

Time ----->

Message M1
AVAILABLE --> delivered to C1 --> invisible for 30s

t0         t1                    t2
|----------|---------------------|

If C1 acks before t2
   -> delete

If C1 crashes and no ack by t2
   -> visible again
+----------------------+
| Message M1           |
| state = INFLIGHT     |
| visible_at = 10:05   |
| leased_to = C1       |
+----------------------+
Recall

  • a state field?

  • all of the above?
Q21If a consumer takes longer than a timeout, what happens?＋
C1 still processing

heartbeat / touch / extend lease
Q23If visibility timeout is too short, duplicates rise. If too long, stuck messages recover slowly. How do you choose it?＋
Open question — reason it out before moving on.
Q24Should each message have the same timeout, or should the timeout be dynamic by job type?＋
Open question — reason it out before moving on.
Engineering Philosophy

The visibility timeout is the queue's heartbeat. Too short and you get duplicate processing. Too long and a crashed consumer holds work hostage. Tune it to your worst-case processing time, not your average.

PROBLEM 06

Ack / delete is deeper than it looks

▶

After consumer finishes work:

Consumer --> DeleteMessage

Sounds simple. But real question is:

  • What exactly is being deleted?
  • How do we know this consumer still owns it?
  • What if delete arrives after timeout?
Consumer C1 gets M1
   |
   v
processes M1
   |
   v
DeleteMessage(receipt_handle)

Need ownership token:

Receipt_handle = proof that this receive attempt owns message

Why receipt_handle is needed

When a consumer gets a message, queue gives not only the message, but also a special temporary token.

ReceiveMessage()
   ->
message = M1
receipt_handle = R123

This token means: “Right now, this consumer is the valid owner of this delivery of M1”.

Why not delete by only message_id?

Because same message can be delivered again later.

Step 1:
C1 gets M1
receipt_handle = R123

Step 2:
C1 is slow
visibility timeout expires

Step 3:
Queue gives M1 again to C2
receipt_handle = R999

Now message M1 has been delivered twice. If old consumer C1 later says DeleteMessage(M1), that is dangerous.

PROBLEM 07

What delivery guarantee do we really provide?

▶

Possible guarantees: at-most-once, at-least-once.

For SQS-like systems, common reality is:

At-least-once delivery
possible duplicates
consumer should be idempotent

Guarantee sketch

Producer write acknowledged
   -> Message stored durably
   -> Consumer receives
   -> Consumer crashes before delete
Message comes again. So duplicates are natural.
Recall
Q29Which is worse for your use case?＋
  • duplicate delivery?
  • lost message?
  • delayed delivery?
Q30Can a queue give exactly-once delivery across network, storage, and consumer crash boundaries?＋
Very hard.
Q31Should the queue guarantee dedup, or should applications handle idempotency?＋
Open question — reason it out before moving on.
Q32For money transfer messages, is at-least-once alone acceptable?＋
Probably not without strong idempotency.
Engineering Philosophy

At-least-once is not a bug — it is a design choice that puts the responsibility for handling duplicates on the consumer. Every service reading from this queue needs to be built with that in mind.

Failure map

Failure can happen anywhere.

visibility timeout expires

==> M1 comes again

This is a normal retry model.

Case 3 — Leader crashes after acknowledging producer

Leader says success
then crashes

==> followers may or may not have message

Case 4 — Delete succeeds on one node but not replicated fully

message considered deleted?
or resurrected later?

PROBLEM 16

What is the source of truth?

▶

Is source of truth:

  • leader memory?
  • leader disk?
  • replicated log?
  • message index DB?
  • lease table?

Source of truth should be:

durable replicated partition log / storage
+
lease/visibility metadata derived or persisted safely
Source of truth = the place we trust most to know whether a message really exists.

That is why we choose: durable replicated log / storage.

Why

  • durable → survives crash
  • replicated → exists on multiple machines
  • rebuildable → index/lease info can be recreated from it

Not these

  • leader memory → lost on crash
  • leader disk only → only one machine copy
  • message index DB → mainly for fast lookup
  • lease table → only says who temporarily owns the message
real truth = stored message
index      = helper
lease      = temporary ownership

IMPLEMENTATION

SQL implementation

▶

One server (usually called partitioning, not true sharding)

  • PostgreSQL — table partitioning inside one database instance.
  • MySQL (InnoDB/NDB engine support) — table partitioning on a single server setup.

Distributed servers (true sharding / distributed SQL)

  • MySQL NDB Cluster — automatically partitions/shards tables across multiple nodes.
  • PostgreSQL + Citus — distributed tables sharded across many PostgreSQL nodes.
  • CockroachDB — distributed SQL database that splits data into ranges/shards across cluster nodes.

Best mental model

One server  -> partitioning
Many servers -> sharding

Also, in practice, when people say “sharding in one server”, they usually really mean partitioning.

In one server partitioning, replicas are usually not between partitions inside that same server. Think like this:

One server
  |
  +--> Partition A
  +--> Partition B
  +--> Partition C

All partitions live on the same DB server. If you want replicas, normally they are made at the database/server level, not “partition A has its own separate replica inside same server”.

Primary server
  |
  +--> all partitions

Replica server
  |
  +--> copy of all partitions

Very short meaning

  • Partitioning = split data for manageability/performance
  • Replication = copy data to another server for HA/read scale
So in one-server partitioning, partitions share the same server, and replica is usually of the whole database/server, not each partition separately.
