FOUNDATION

Core idea

▶

This is one of those systems that looks simple at first: append data to a log. But once you go deeper, it becomes a very rich design problem:

writes
reads
compaction
crash recovery
indexing
deletes
space amplification
read amplification
merge policy

In a log-structured storage system, instead of updating data in place, we mostly do:

old version stays
new version is appended

So storage becomes like a growing sequence of records.

Main mental model

Traditional update:
go to old place and overwrite

Log-structured update:
append new version at end
later clean old versions

Why people use it

Because appends are usually good for:

  • high write throughput
  • crash recovery simplicity
  • immutable segment design
FOUNDATION

What is log-structured storage?

▶

In-place update model

Disk / file
+----+----+----+----+----+----+
| A  | B  | C  | D  | E  | F  |
+----+----+----+----+----+----+

Update C
+----+----+----+----+----+----+
| A  | B  | C' | D  | E  | F  |
+----+----+----+----+----+----+

We go back and overwrite.

Log-structured model

Initial log
+----+----+----+----+
| A  | B  | C  | D  |
+----+----+----+----+

Update C
+----+----+----+----+----+
| A  | B  | C  | D  | C' |
+----+----+----+----+----+

Old C remains. New C' is appended.

Another update
+----+----+----+----+----+----+
| A  | B  | C  | D  | C' | C''|
+----+----+----+----+----+----+
Now the latest value of C is C''.
Engineering Philosophy

Log-structured storage makes writes fast by making reads and compaction do more work. Every design decision in an LSM engine flows from this single trade-off.

PROBLEM 01

If old versions remain, how do we know the latest one?

▶

Log

offset 10 -> key=A value=1
offset 20 -> key=B value=7
offset 30 -> key=A value=2
offset 40 -> key=C value=9
offset 50 -> key=A value=3

What is the latest value of A? The one with highest valid offset / newest sequence:

A = 3

But scanning full log is bad

Read A
--> scan log
--> find A at 10
--> find A at 30
--> find A at 50
--> choose latest

Works conceptually, terrible for performance.

So we build an index

In-memory index

A -> offset 50
B -> offset 20
C -> offset 40
Read path:
key A
 |
 v
mem index says offset 50
 |
 v
read record at offset 50

PROBLEM 02

If appending is easy, what happens to old stale data?

▶

Because append-only creates garbage.

write A=1
write A=2
write A=3
write A=4

Log becomes:
+-------------------------------+
| A=1 | A=2 | A=3 | A=4         |
+-------------------------------+

Only latest value matters for current state:

A=4 is live
A=1 A=2 A=3 are stale

Storage picture

Segment 1:
+-----------------------------+
| A=1 | B=1 | C=1 | D=1       |
+-----------------------------+

Segment 2:
+-----------------------------+
| A=2 | C=2 | E=1 | F=1       |
+-----------------------------+

Segment 3:
+-----------------------------+
| A=3 | B=2 | G=1 | H=1       |
+-----------------------------+

Current live values might be:

A -> from seg3
B -> from seg3
C -> from seg2
D -> from seg1
E -> from seg2
F -> from seg2
G -> from seg3
H -> from seg3

Many old entries are dead.

So we need compaction / cleaning

Old segments
   |
   v
read them
   |
   v
keep only latest live records
   |
   v
write new compacted segments

Compaction sketch

Before compaction

Seg1: [A1][B1][C1][D1]
Seg2: [A2][C2][E1][F1]
Seg3: [A3][B2][G1][H1]

Latest live:
A3 B2 C2 D1 E1 F1 G1 H1
After compaction

Seg4: [A3][B2][C2][D1][E1][F1][G1][H1]

Old segments can be deleted.

Questions
Q6If compaction removes old versions, is the system still truly append-only?＋

Interesting answer:

write path        = append-only
storage lifecycle = append + rewrite + delete old segments
Q7What is the real tradeoff?＋
fast writes now
cleaning cost later
Q8If workload is update-heavy, what happens?＋
same key rewritten many times
garbage grows quickly
compaction pressure rises
Q9If compaction is too aggressive, what happens?＋
high rewrite cost
CPU cost
I/O contention
write amplification
Q10If compaction is too lazy, what happens?＋
wasted space
slower reads
longer restart rebuild
worse cache efficiency
Engineering Philosophy

Compaction isn't housekeeping — it's how the system stays correct. If compaction falls behind, reads get slower and storage grows. Compaction lag is an early warning sign.

PROBLEM 03

How do deletes work in append-only storage?

▶

In in-place storage, delete may mean “remove or mark deleted at same place”. In log-structured storage, we usually append a tombstone.

Delete sketch

write A=100
write B=200
delete A

Log:
+----------------------------------+
| A=100 | B=200 | A=TOMBSTONE      |
+----------------------------------+
Meaning: latest state of A = deleted.

Yes:

That is a clean mental model.

PROBLEM 04

How do we write safely without corrupting state on a crash?

▶

One major strength of log-structured systems is that crash handling can be elegant, because append has nice properties.

Write flow

client write
   |
   v
append record to active log segment
   |
   v
flush / fsync as required
   |
   v
update in-memory index

But ordering matters.

Important crash scenario

Wrong order
1. update index in memory
2. crash before log persisted

After restart, index pointed to data that never became durable. Bad.

Better order
1. append to log buffer
2. persist / checksum / fsync
3. then expose as committed
4. update in-memory structures

Crash recovery sketch

Log segment bytes:
[record1][record2][record3][partial garbage....crash]

Recovery:
scan until last valid checksum / length boundary
ignore torn tail
RecapSay why ordering matters in your own words — then open.＋

In a log-structured storage system, the log must become durable first, and only after that should the system behave as if the write is real. Why? Because memory can lie after a crash.

If you update the in-memory index first and the machine crashes before the log record is actually flushed to disk, then after restart your system may think the data exists, but the durable storage does not contain it. That means the index was pointing to something that was never safely saved. This creates corruption or inconsistency.

So the safe idea is:

write to log -> persist it properly -> mark it committed / visible -> update memory structures

This ordering ensures that whatever the system exposes as written already exists in durable form.

In recovery, the system reads the log from the start and trusts only the part that is fully valid. If the crash happened in the middle of writing the last record, the tail may contain partial or torn bytes. Using record length and checksum, recovery can detect where the last good record ends and simply ignore the broken tail. So append-only logs make crash recovery simpler, as long as write ordering is correct.

Questions
Q16What is the actual source of truth after crash?＋
the durable log
not the in-memory index
Q17Can every in-memory structure be rebuilt from durable log?＋
If yes, design is cleaner.
Q18Do you need checksum per record, per block, or per segment?＋
Open question — reason it out before moving on.
Q19What happens if crash occurs in middle of a record?＋

Need record framing:

[length][key][value][checksum]
Q20Is fsync after every write necessary?＋
safer
slower

So often we batch.

Engineering Philosophy

A write that hasn't hit durable storage isn't really a write. The WAL is the line between what you promised your caller and what you can actually survive a crash with.

PROBLEM 05

If segments are immutable, how do reads stay fast?

▶

Append-only alone is not enough. Fast reads require structure.

Segment model

Instead of one giant infinite log, usually we use segments.

Segment 1
Segment 2
Segment 3
Segment 4 (active write segment)

old segments   = immutable
current segment = appendable
+---------+  +---------+  +---------+  +---------+
| Seg 1   |  | Seg 2   |  | Seg 3   |  | Seg 4   |
| sealed  |  | sealed  |  | sealed  |  | active  |
+---------+  +---------+  +---------+  +---------+

This gives many benefits:

  • easier compaction
  • easier crash recovery
  • easier indexing
  • easier file lifecycle

Read lookup options

memtable / sparse index / bloom filter / block index

PROBLEM 06

What exactly is being compacted?

▶

Compaction is not just “merge files”. It is:

determine live records
drop stale versions
preserve newest visible state
preserve delete semantics
rewrite into new clean segments
Input segments:
Seg1: [A1][B1][C1]
Seg2: [A2][D1][E1]
Seg3: [B2][A3][F1]

Choose newest per key:
A3 B2 C1 D1 E1 F1

Write compacted output:
Seg4: [A3][B2][C1][D1][E1][F1]

How do we know A3 is newer than A2?

Need version ordering by one of these:

- log position
- sequence number
- timestamp
- hybrid logical clock
- version vector in distributed system

These are not always equivalent.

Yes. Very dangerous in distributed systems.

PROBLEM 07

Write vs read vs space amplification

▶

This is one of the deepest architect-level tradeoff zones.

In log-structured storage, data is first appended quickly, but later the system has to manage old and new versions, and that creates three kinds of extra cost.

1. Write amplification

One small user write may be rewritten many times during compaction, so the system writes much more data internally than the user actually sent.

user wrote 1 MB
system eventually writes 4 MB total internally

2. Read amplification

Reading one key may require checking multiple places, because the latest value could be in memory, a newer segment, or an older segment.

read key K
check memtable
check recent segment
check older segment
check disk block

3. Space amplification

Old stale data stays on disk until compaction cleans it up, so physical storage used can be much larger than the actual live data.

logical live data = 100 GB
physical stored   = 300 GB

Triangle sketch

                +------------------+
                |  Write Amplify   |
                +------------------+
                   /            \
                  /              \
                 /                \
                v                  v
      +----------------+    +----------------+
      | Read Amplify   |    | Space Amplify  |
      +----------------+    +----------------+
Usually reducing one hurts another.
Questions
Q31If I compact aggressively, what happens?＋
space improves
reads improve
writes worsen
Q32If I compact lazily, what happens?＋
writes improve
space worsens
reads may worsen
Q33Which amplification matters most for my workload?＋
  • write-heavy ingest store?
  • read-heavy KV store?
  • archival store?
  • SSD vs HDD?
  • cloud object storage?
Engineering Philosophy

Write amplification, read amplification, and space amplification cannot all be minimized at once. Choosing a storage engine means choosing which one you can live with.

PROBLEM 08

How do we design the write path?

▶

Common write path

Client write
   |
   v
Memtable
   |
   v
Append to WAL / active segment
   |
   v
Ack after durability policy
   |
   v
Later flush buffer to immutable segment

Depending on the system, WAL and segment may be same or separate.

             +-------------------+
write -----> | memory buffer     |
             +---------+---------+
                       |
                       v
             +-------------------+
             | append log        |
             +---------+---------+
                       |
                       v
             +-------------------+
             | fsync / durable   |
             +-------------------+
RecapSay the simple view in your own words — then open.＋

When a write comes, the system first updates the in-memory hashmap / memtable for fast access, but memory can be lost on crash, so before saying “write successful,” it also appends the write to disk in a crash-safe way. That disk append is called WAL (Write-Ahead Log) or active segment.

“fsync / durable” means forcing that appended log record to actually reach a stable disk, not just stay in the OS cache.

Questions

Because:

  • fewer disk writes
  • batching
  • sorted flush possible
  • better throughput
Q36Should acknowledgment happen after memory only, OS page cache, fsync, or replication quorum?＋
This changes durability semantics.
Q37If system acknowledges before true durability, what failures are acceptable?＋
Open question — reason it out before moving on.
Q38Does buffer preserve write order? Should it?＋
Open question — reason it out before moving on.

PROBLEM 10

What happens on startup / restart?

▶

Restart picture

System crashes
memory gone
disk segments remain

Need to restore:

latest index
active segment position
valid committed tail
compaction state if any

Recovery sketch

On startup:
1. find segment files
2. scan metadata / manifests
3. rebuild or load indexes
4. detect incomplete tail
5. resume service

Two design styles

A. Rebuild from data

scan log / segments
reconstruct index

Simple, but maybe slow.

B. Keep checkpoints / manifests

load saved metadata
do small delta replay

Faster, more metadata complexity.

PROBLEM 12

Is this just a log, or does it become LSM-like?

▶

Pure log-structured storage can mean raw append history. But many practical systems evolve toward something like:

write fast like a log
organize immutable sorted segments
compact/merge in background

That is why log-structured storage and LSM-style storage are conceptually close in many systems.

Writes -> append
Flush  -> immutable segment
Merge  -> cleaner larger segment
Reads  -> index + filters + segment search

Usually easier for merge logic, but flush path changes.
Q53If segments are unsorted, write path is simpler or faster, but read path/compaction may get harder. Which side do you want to optimize?＋
Open question — reason it out before moving on.
