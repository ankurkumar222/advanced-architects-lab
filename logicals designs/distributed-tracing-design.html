STAGE MAP

Two stages

▶

One user request touches forty services. A trace is the reconstructed causal story of that one request. Splitting into 2 stages makes it much easier to read:

  • Stage 1 = how the trace is produced and shipped without hurting anyone
  • Stage 2 = how it is stored, reassembled and queried

Stage 1 — produce path

  • request enters, root span starts
  • trace context propagates across every hop
  • each service emits spans locally
  • sampling decides what survives
  • SDK buffers, agent batches, collector receives
Request -> Root Span -> Context Propagation -> Child Spans -> Sampler -> Buffer -> Agent -> Collector

Stage 2 — consume path

  • collector normalizes and enriches
  • spans written to store, indexes updated
  • spans of one trace reassembled into a tree
  • user searches, opens waterfall, drills down
Collector -> Queue -> Span Store + Index -> Assembly -> Search / Waterfall / Dependency Map
                                    |
                                    v
                             Retention / Tiering
Core job: reconstruct one request's causal tree from thousands of independently emitted, partially sampled, clock-skewed fragments.
PROBLEM 01

What is a trace really?

▶

Before design, define the object clearly. A trace is not a log line. It is a tree of timed intervals.

Span
----
trace_id
span_id
parent_span_id
service_name
operation_name
start_time
duration
status          (ok / error)
attributes      (key-value tags)
events          (timestamped points inside the span)
links           (references to other traces)

Trace as a tree

trace_id = T1

  [api-gateway  /checkout]                span A (root)
       |
       +-- [order-svc  createOrder]       span B  parent=A
       |         |
       |         +-- [db  INSERT orders]  span D  parent=B
       |
       +-- [payment-svc  charge]          span C  parent=A
                 |
                 +-- [psp  POST /charge]  span E  parent=C

Same trace as a waterfall

time ----------------------------------------->

A  |=========================================|   420ms
B    |==============|                             150ms
D      |=========|                                 90ms
C                  |========================|     260ms
E                    |====================|       210ms
The tree gives you causality. The waterfall gives you blame. You need both views from the same data.
Questions
Q1Is a span one record with a duration, or two events (start and end)?＋
One record:
emit once at end, duration included
simpler storage, but nothing visible while in flight

Two events:
emit start, emit end
live view possible, but 2x volume and orphan risk
Q2Is a trace strictly a tree, or a DAG?＋

A fan-in — one consumer batch triggered by many producers — has multiple causes. That is why links exist separately from parent_span_id.

Q3Who generates trace_id, and what should it be?＋
random 128-bit at first hop
or
derived from request id at edge

Random avoids coordination. But then trace_id carries no routing information — think about whether you want it to.

Q4What if a parent span never arrives?＋

You have orphans. Do you drop them, show a synthetic parent, or render a forest instead of a tree?

Q5Should operation_name be free text?＋
GET /users/1001
GET /users/1002
GET /users/1003

Three operations, or one? This single decision determines whether your aggregation is useful or useless.

PROBLEM 02

Context propagation is the spine

▶

Everything else is decoration. If context does not travel, you have no trace — only disconnected spans.

Two propagation problems, not one

1. In-process
   pass context across function calls, threads, async tasks

2. Cross-process
   pass context across the network boundary

Cross-process sketch

Service A
   |
   | HTTP headers:
   |   traceparent: 00-{trace_id}-{span_id}-{flags}
   |   tracestate:  vendor-specific
   v
Service B
   |
   | reads header -> trace_id kept
   |               -> incoming span_id becomes parent_span_id
   |               -> new span_id generated
   v
Service C

The break

A --header--> B --(header dropped)--> C --header--> D

Result:
trace T1 = { A, B }
trace T2 = { C, D }     <- new root, no link to A

One request, two unrelated traces.
One un-instrumented hop in the middle silently cuts every trace in half. Propagation coverage matters more than instrumentation depth.
Questions
Q6How does context survive an async boundary inside one process?＋
thread-local        -> breaks on thread pool handoff
explicit parameter  -> correct but invasive
async-local storage -> usual answer in modern runtimes
Q7How does context travel through a queue?＋
Producer writes context into message headers
Consumer reads it back

Producer span ends at publish
Consumer span starts at receive

Do you make the consumer a child, or a linked span in a new trace? Wait time in the queue is not the consumer's fault.

Q8A nightly batch reads 10 million rows. One trace or ten million?＋
Open question — reason it out before moving on.
Q9Should propagation be done by library, by service mesh sidecar, or both?＋

Mesh gives free coverage but only sees network hops — it cannot see inside your process. Library sees everything but requires every team to adopt it.

Q10Should you propagate business context (tenant_id, user tier) alongside trace context?＋

Very useful for filtering. But baggage travels on every hop, costs bytes on every request, and can leak sensitive data outside your trust boundary.

PROBLEM 03

Who actually creates spans?

▶

Three sources, three different levels of detail and cost.

1. Auto-instrumentation
   framework hooks: HTTP server, HTTP client, DB driver, queue client
   free coverage, generic naming

2. Manual instrumentation
   developer wraps a meaningful business step
   best signal, requires discipline

3. Infrastructure
   mesh proxy, load balancer, gateway
   no code change, but only edge-level view
                +---------------------------+
                |  one incoming request     |
                +-------------+-------------+
                              |
      +-----------------------+----------------------+
      |                       |                      |
      v                       v                      v
 mesh span              framework span         business span
 (proxy in/out)         (HTTP handler)         (validateLimit)
 free, coarse           free, generic          costly, precious
Questions
Q11What is the actual overhead of a span?＋
allocation
timestamp read
attribute map
serialization
buffer append
network export

At 50k requests/sec with 20 spans each, that is 1M spans/sec from one service. Do the arithmetic before deciding instrumentation depth.

Q12Should a span exist for every function, or only for boundaries?＋

Boundaries — network, disk, lock, external call — are where time actually goes. Function-level spans usually produce noise, not insight.

Q13Who owns naming consistency across 80 microservices?＋

If order-svc calls it createOrder and order-service calls it POST /orders, no aggregation across the fleet is possible. This is a platform governance problem disguised as a technical one.

Q14Should errors be a status field, a span event, or both?＋
Open question — reason it out before moving on.
PROBLEM 04

Clocks lie

▶

Spans come from different machines. Machine clocks disagree by milliseconds — sometimes seconds.

Reality:
A calls B, so B must start after A

Recorded:
A.start = 10:00:00.500   (host 1)
B.start = 10:00:00.480   (host 2, clock 40ms behind)

Waterfall renders child starting before parent.
Duration appears negative.
What you can trust:
  duration within one process   -> monotonic clock, reliable
  causal order                  -> parent/child relationship, reliable

What you cannot trust:
  absolute timestamps across hosts
  cross-host subtraction (network latency estimates)
Questions
Q15Should you record wall-clock time, monotonic duration, or both?＋

Both. Wall clock for ordering and search; monotonic for the duration itself, so duration never goes negative even if NTP steps the clock mid-span.

Q16Do you correct skew at write time or at read time?＋
Write time:  stored data is "clean" but lies about what was observed
Read time:   raw data preserved, renderer clamps child inside parent

Read time is usually safer — you can change the correction later.

Q17Can you compute network latency as (child.start − parent.start)?＋

Not reliably. That number contains clock skew plus queueing plus serialization. Treat it as a hint, never as a metric.

Q18Is the trace timeline the parent's clock or a global clock?＋

Anchoring everything to the root span's clock and expressing children as offsets makes the waterfall coherent, even if the absolute times are wrong.

PROBLEM 05

Sampling is the real economics

▶

You cannot store every trace. The whole design question is which traces are worth keeping — and when you decide.

Head-based sampling

Decide at the root, propagate the decision in the flag bit.

Root span
   |
   | roll dice: keep 1%
   | sampled = true
   v
propagate flag in traceparent
   |
   +--> B honours flag -> emits
   +--> C honours flag -> emits
Cheap, stateless, consistent — every service makes the same decision. But you decide before you know whether the request was interesting.

Tail-based sampling

Buffer all spans of a trace, decide when the trace is complete.

all spans -> collector buffer (keyed by trace_id)
                    |
              wait for trace to settle
                    |
         +----------+-----------+
         |                      |
      had error?             was slow?
      keep 100%              keep 100%
                    |
              otherwise keep 1%
You keep exactly the traces you needed. But the collector must hold every span of every in-flight trace in memory, and all spans of one trace must land on the same collector instance.
Questions
Q19If you sample 1% at head, how do you ever catch the rare error?＋

You mostly do not. That is the central argument for tail sampling — the failures you need are, by definition, rare.

Q20Tail sampling needs all spans of a trace in one place. How do you route them there?＋
hash(trace_id) -> collector shard

Now collectors are stateful.
Rebalancing them splits in-flight traces.
Q21How long do you buffer before declaring a trace complete?＋
too short -> long traces get truncated, decision made on partial data
too long  -> memory grows, decision latency grows

This is the same shape as visibility timeout in a queue: a guess about how long work takes.

Q22Should the sampling rate be per service, per endpoint, or per tenant?＋

A flat global rate over-samples your busiest endpoint and under-samples the one that runs twice an hour. Rate-limiting per operation gives better coverage than probability.

Q23If sampling is biased toward errors, are your latency percentiles still valid?＋

No. Sampled traces are for debugging. Percentiles must come from unsampled aggregates computed before the sampler — see problem 14.

Q24Can head and tail sampling coexist?＋

Often yes — a generous head sample to bound volume at source, then a selective tail sample to decide what is stored.

PROBLEM 06

The ingest pipeline

▶

Between the application and the database there are four buffers, and every one of them can drop your data.

App process
   |
   |  in-process SDK buffer          (bounded queue)
   v
Agent / sidecar
   |
   |  batching + compression         (per-node buffer)
   v
Collector
   |
   |  normalize, enrich, tail-sample (stateful)
   v
Durable queue
   |
   |  decouple ingest from storage
   v
Span store + index writers

Where each buffer helps

SDK buffer     -> absorbs export latency, keeps request path fast
Agent          -> batches many processes on one host, one connection out
Collector      -> central policy: sampling, redaction, enrichment
Durable queue  -> storage can be slow or down without losing spans
Questions
Q25When the SDK buffer is full, what happens?＋
block the request thread   -> tracing has now caused an outage
grow unbounded             -> OOM, tracing has now caused an outage
drop spans + count drops   -> correct answer
Q26Should you drop oldest or newest when dropping?＋
Open question — reason it out before moving on.
Q27Should partial traces be shipped, or only complete ones?＋

The SDK cannot know when a trace is complete — it only sees its own process. Completeness is a collector-side concept.

Q28Where does PII get stripped?＋

SDK-side redaction is safest — the data never leaves the process. Collector-side is easier to update centrally. Most teams need both.

Q29Does the collector need to be stateless?＋

It would like to be. Tail sampling takes that away — see Q20.

Q30What is the durability boundary? When is a span "safe"?＋

Nobody acknowledges a span back to the application. Tracing is fire-and-forget by design — which means every buffer before the durable queue is a lossy hop you accepted on purpose.

PROBLEM 07

How should spans be stored?

▶

Two completely different access patterns, so usually two storage shapes.

Access pattern 1 — "show me trace T1"
   exact key lookup, return 40 spans

Access pattern 2 — "slow checkout traces in the last hour"
   search by service + operation + time + tags, return trace_ids

Two-store model

span batch
   |
   +--> SPAN STORE
   |      key: trace_id (+ span_id)
   |      value: raw span blob
   |      write-once, read-by-key
   |
   +--> INDEX STORE
          key: service + operation + time bucket
          value: trace_id, duration, status, indexed tags
SPAN STORE partitioned by hash(trace_id)
   good: all spans of a trace land together -> assembly is one shard read
   bad:  no time locality, retention deletes are scattered

INDEX STORE partitioned by time + service
   good: searches are bounded, old partitions dropped whole
   bad:  hot partition on the busiest service in the current minute
Questions
Q31Should the partition key be trace_id or time+service?＋

Those two answers optimize opposite queries. That is precisely why most systems run both stores rather than choosing.

Q32Is a span ever updated after write?＋

Effectively never. Spans are immutable facts. That makes append-only, LSM-style storage a natural fit and makes retention a matter of dropping whole segments.

Q33Should the raw span be stored as a blob or as columns?＋
Blob:    cheap write, whole-span read, no analytics
Columnar: expensive write, but "p99 duration by operation" becomes a scan
Q34If the index write succeeds but the span write fails, what does the user see?＋

A search result that opens into an empty trace. Which of the two writes should happen first, and should the reader tolerate the gap?

Q35One trace can span multiple time buckets. Which bucket owns it?＋
Open question — reason it out before moving on.
PROBLEM 08

Trace assembly is the hard read

▶

Spans of one trace are emitted by different processes, at different times, over different networks. They arrive out of order, late, or not at all.

Emission order (real time):
D ends first (90ms db call)
B ends next
E ends
C ends
A ends last  (root always finishes last)

So the root — the thing you need to build the tree — arrives LAST.
Arrival at storage:
t+0.1s   D
t+0.2s   B
t+0.4s   E
t+0.5s   C
t+2.0s   A        <- retried export, agent restart, slow host
t+45s    F        <- straggler from a lagging service

Query at t+1.0s renders a broken forest.

Two assembly strategies

A. Assemble at read time
   fetch all spans by trace_id, build tree in memory
   always reflects latest data, cost paid per query

B. Assemble at write time
   buffer, wait a window, materialize the tree once
   fast reads, but stragglers arrive after the tree is sealed
Questions
Q36When is a trace "complete"?＋

There is no completion signal. Nobody knows how many spans a trace should have. Completeness is always a timeout, never a fact.

Q37Could the root span carry an expected child count?＋

The root does not know — children are created deep in services it never sees. This is why tracing systems are eventually consistent by nature.

Q38If a span arrives after the trace was rendered, do you re-render?＋
Open question — reason it out before moving on.
Q39How do you display a trace whose root is missing?＋
synthetic root spanning min(start) to max(end)
+ explicit "incomplete trace" marker

Silently drawing a partial tree as if it were whole is how people misdiagnose incidents.

Q40Should the UI show what is missing, or only what is present?＋

A gap in a waterfall usually means an un-instrumented hop. Making the gap visible turns a blind spot into a to-do item.

PROBLEM 09

What do people actually query?

▶

Design the index around the real questions, not around the data model.

Q-type 1  "open trace abc123"                  key lookup
Q-type 2  "slow checkouts, last 1h, p99"       time + service + duration filter
Q-type 3  "traces with error in payment-svc"   time + service + status filter
Q-type 4  "traces for order_id = 88213"        high-cardinality tag lookup
Q-type 5  "what does order-svc depend on?"     aggregate over edges
Q-type 6  "did latency change after deploy?"   compare two time windows
Type 1  -> span store, one shard
Type 2  -> index store, range scan on (service, time, duration)
Type 3  -> index store, same shape, status filter
Type 4  -> needs a tag index, and that is where cardinality bites
Type 5  -> precomputed dependency graph, not a query over raw spans
Type 6  -> precomputed aggregates, not sampled traces
Types 5 and 6 should never be answered by scanning stored traces. They are stream-derived, not search-derived.
Questions
Q41Does search return traces or spans?＋

The user thinks in traces; the index holds spans. A trace matches if any span matches — so the index returns span hits and you must deduplicate by trace_id.

Q42"Slow trace" means slow root, or contains a slow span?＋

Very different queries. A fast trace can contain a slow parallel branch that nobody waited on.

Q43Should root-span duration be denormalized onto every child?＋

It makes "slow trace" searchable from any span, at the cost of writing a value the child does not yet know — the root has not finished when the child is emitted.

Q44Should the system support full-text search inside span attributes?＋
Open question — reason it out before moving on.
PROBLEM 10

Cardinality is the silent killer

▶

The attributes that make a trace useful for debugging are exactly the ones that destroy your index.

Low cardinality — safe to index
   service_name      ~80 values
   http.status_code  ~40 values
   region            ~5 values

High cardinality — precious but expensive
   user_id           millions
   order_id          billions
   session_id        billions
   sql.query_text    unbounded
Index every attribute:
   index size approaches data size
   write amplification per span explodes
   one badly-named tag can take down ingest

Usual compromise

allow-list of indexed attributes   -> cheap, predictable
everything else stored but not indexed
+ separate inverted index for a few chosen high-cardinality keys
  (order_id, trace-linking ids) with shorter retention
Questions
Q45Allow-list or deny-list for indexed attributes?＋

Deny-list means an application team can break the platform by adding one tag. Allow-list means they must ask you first. Which failure do you prefer?

Q46Should high-cardinality indexes have shorter retention than spans?＋

Usually yes. "Find the trace for this order" is an urgent question asked within hours, not months.

Q47Should operation_name be normalized at ingest?＋
GET /users/1001  ->  GET /users/{id}

Without this, operation_name is high-cardinality and every aggregation is meaningless. With it, you lose the specific path unless you also keep it as an attribute.

Q48Who is accountable when one service's tags blow up the index?＋

This is problem 13 in disguise — a fairness question, not a storage question.

PROBLEM 11

Retention, tiering, cost

▶

Trace data is enormous, and its value decays faster than almost any other data you store.

Value curve:
0 - 2 hours    incident debugging          very high
2 - 48 hours   post-incident review        high
2 - 30 days    regression comparison       medium
30 days +      compliance / audit only     low
Tier 1  hot     0-2 days     fast store, fully indexed
Tier 2  warm    2-14 days    cheaper store, reduced index
Tier 3  cold    14-90 days   object storage, error traces only
Tier 4  gone    beyond that  aggregates survive, traces do not
Questions
Q49Should retention be uniform, or per service / per tenant?＋
Open question — reason it out before moving on.
Q50Should error traces live longer than successful ones?＋

Almost always yes — and this argues for the sampling decision being recorded on the span, so retention can act on it later.

Q51Is deletion a per-span operation or a segment drop?＋

Per-span deletion at this volume is unaffordable. Time-partitioned segments that can be dropped whole is the only design that scales — which constrains your partitioning choice back in problem 7.

Q52If a trace is 1% sampled, what did you actually pay for the other 99%?＋

CPU in the app, bytes on the wire, and collector memory — all spent on data you then discarded. Head sampling exists to avoid exactly that cost; tail sampling accepts it deliberately.

PROBLEM 12

Tracing must never hurt the app

▶
The single non-negotiable rule: an observability system that can take down production is worse than no observability system.
Failure map
App -> SDK buffer -> Agent -> Collector -> Queue -> Storage

Case 1 — Collector unreachable

agent retries
buffer fills
==> must drop, never block

Case 2 — Collector slow, not down

export takes 5s instead of 50ms
if export is synchronous, every request pays 5s

==> export must always be off the request path

Case 3 — A service emits 100x normal spans

bad deploy adds a span in a tight loop
one service saturates the whole pipeline

==> need per-source rate limits at the agent, not just at the collector

Case 4 — Storage down

durable queue absorbs it
==> how deep, and what happens when the queue also fills?
Questions
Q53Should tracing failures be visible, or silent?＋

Silent dropping is how teams end up debugging an incident with a trace that quietly lost half its spans. Emit a dropped-span counter as a metric — tracing needs its own observability.

Q54Should there be a kill switch?＋

A remotely-settable sample rate that can go to zero, taking effect without a redeploy, is the difference between a bad hour and a bad day.

Q55What is the memory ceiling of the SDK, and who enforces it?＋
Open question — reason it out before moving on.
Q56Does the tracing system trace itself?＋

Tempting, and a beautiful way to build an infinite loop. Usually the pipeline is monitored by metrics, not by its own traces.

PROBLEM 13

Multi-tenant fairness

▶

Whether tenants are external customers or internal teams, the shape is the same.

Team A (payments)   50,000 spans/sec
Team B (reporting)      200 spans/sec
Team C (onboarding)   1,200 spans/sec
                 +-----------------------+
                 | Shared Trace Pipeline |
                 +-----------+-----------+
                             |
          +------------------+------------------+
          |                  |                  |
          v                  v                  v
       Team A             Team B             Team C

Without controls, A's bad deploy erases B's and C's traces.

Need isolation:

  • per-tenant ingest quotas
  • per-tenant sampling policy
  • per-tenant index and storage budgets
  • separate queues or partitions for the loudest tenants
Questions
Q57Should quota enforcement be at the agent, the collector, or both?＋

Agent-side is cheaper — the data never crosses the network. Collector-side is authoritative. Both, with the agent as the fast path.

Q58When a tenant exceeds quota, do you drop or downsample?＋

Dropping loses whole traces at random — usually worse than lowering the sample rate, which keeps whole traces and loses others cleanly.

Q59Can one team's traces reveal another team's data?＋

A trace crosses service boundaries by definition, so it crosses team boundaries too. Who is allowed to see the payment span inside the checkout trace?

PROBLEM 14

Derived signals from traces

▶

The most valuable outputs of a tracing system are usually not traces.

span stream (pre-sampling)
   |
   +--> RED metrics       rate, errors, duration per operation
   |
   +--> service graph     edges derived from parent.service -> child.service
   |
   +--> SLO / alerting    error ratio per endpoint
   |
   +--> exemplars         link a p99 metric point to one real trace
Compute these from the unsampled stream. If you compute them after sampling, every number is biased by your sampling policy.

Service graph from spans

for each span with a parent:
    edge = (parent.service_name -> span.service_name)
    increment count, error count, latency histogram

api-gateway ---> order-svc ---> postgres
      |                \
      |                 +-----> inventory-svc
      +--------> payment-svc ---> psp-external
Questions
Q60Should metrics be derived from spans, or emitted independently?＋
From spans:      one instrumentation, guaranteed consistency
Independent:     survives tracing outage, cheaper, no sampling bias

Most mature setups do both and reconcile — which raises the question of which one is authoritative during an incident.

Q61The service graph shows an edge that no architecture diagram has. Who is wrong?＋

The trace. Discovering undocumented dependencies is often the highest-value output of the whole system.

Q62An exemplar links a metric spike to a stored trace — but the trace was sampled away. Now what?＋

Either the exemplar forces that trace to be retained (sampling decision made by the metrics pipeline), or your exemplar is a dead link. This couples two systems that look independent.

PROBLEM 15

Span lifecycle is the soul

▶
          +-----------+
          | STARTED   |   context created, clock read
          +-----+-----+
                |
                v
          +-----------+
          |  ENDED    |   duration fixed, status set
          +-----+-----+
                |
                v
          +-----------+
          | BUFFERED  |   in SDK queue
          +--+-----+--+
             |     |
      export |     | buffer full
             v     v
        +----------+   +-----------+
        | EXPORTED |   |  DROPPED  |
        +----+-----+   +-----------+
             |
             v
        +----------+
        | SAMPLED  |   collector keeps or discards
        +----+-----+
             |
             v
        +----------+
        | INDEXED  |   searchable
        +----+-----+
             |
             v
        +----------+
        | EXPIRED  |   retention drops the segment
        +----------+
Every arrow in this diagram is a place your data can silently disappear. Count them: there are six.

If I reduce everything to only the hardest core

start a span
-> propagate context across every hop
-> decide early or late what is worth keeping
-> ship it without ever blocking the request
-> store it twice, by id and by search key
-> reassemble a tree that has no completion signal
-> answer the question before the incident ends
-> and never, ever take production down doing it
