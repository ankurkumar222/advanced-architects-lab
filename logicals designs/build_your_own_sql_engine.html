The surface illusion

▶

From the outside, a SQL database looks deceptively simple:

App --> SELECT / INSERT / UPDATE / DELETE
But this abstraction masks a profound intellectual challenge: How do you build a system that stores relational data, answers expressive queries, and preserves correctness under concurrent updates?

PROBLEM 01

The relational revolution

▶

What makes SQL DB fundamentally different from NoSQL?

The deepest distinction isn't syntax — it's the data model.

SQL DB is built around:

  • Relations (tables) with structured schemas
  • Joins that connect entities naturally
  • Constraints that preserve data integrity
  • Transactions that guarantee correctness
  • Declarative querying where you say what, not how

Thought experiment

Customers                      Orders
+----+--------+               +----+-------------+--------+
| id | name   |               | id | customer_id | total  |
+----+--------+               +----+-------------+--------+
| 1  | Ankur  |               | 10 | 1           | 500    |
| 2  | Ravi   |               | 11 | 1           | 900    |
+----+--------+               +----+-------------+--------+

SQL naturally asks: join customers and orders, filter, aggregate, sort, group. This is much richer than pure key-value lookup.

PROBLEM 02

Constraints: the silent guardians

▶

A SQL DB is not just rows — it's enforced invariants:

  • Primary key
  • Foreign key
  • Unique
  • Not null
  • Check constraints
Orders.customer_id MUST reference Customers.id
Key insightWhy does enforcement in the database change how you think about it?＋
The database is active in preserving correctness, not just passively storing data. This is both a strength and a constraint — it guarantees integrity but limits flexibility.
PROBLEM 03–04

The storage foundation

▶

How is relational data actually stored?

Page-oriented storage

Internally, SQL DB stores data in fixed-size pages (typically 4KB–16KB):

Disk
  |
  +--> Page 1
  +--> Page 2
  +--> Page 3
  +--> Page 4

A table is not “just a list” — it's pages containing rows:

Table: orders
Page 101: [row][row][row]
Page 102: [row][row][row]
Page 103: [row][row][row]

Two storage models

Heap styleClustered / index-organized
Rows stored wherever free space existsRows physically ordered by primary/cluster key
Insertion is fast but scans are scatteredInserts maintain order but can be expensive
Secondary indexes point to row locationsSecondary indexes point to primary key
Tradeoff: heap is better for high-frequency inserts; clustered is better for range scans and ordered access.
PROBLEM 05–07

Indexes: the performance soul

▶

Indexes are one of the main hearts of a SQL DB. Without them, queries become table scans.

The classic B+ tree

               [50]
              /      \
          [20]        [80]
         /   \       /    \
      ...   ...    ...    ...

Leaf nodes point to row locations or contain actual row data.

Why B+ tree is central

  • Balanced (consistent performance)
  • Ordered (range scans possible)
  • Good for point lookup, range scans, sort-related access

Composite indexes — problem #6

Consider this query:

SELECT *
FROM orders
WHERE customer_id = 1001
ORDER BY created_at DESC;

The optimal index might be:

(customer_id, created_at)

Index entries ordered like:

customer=1001, time=2026-04-01
customer=1001, time=2026-04-02
customer=1001, time=2026-04-03
customer=1002, time=2026-04-01

Now the database can directly seek to customer 1001 and scan ordered by time.

Covering indexes — problem #7

Sometimes an index contains enough columns that the query can finish without touching the main table:

-- Index stores (department, name, salary)
SELECT name, salary
FROM employees
WHERE department = 'Engineering';
The query is answered entirely from the index — no table access needed. This is a massive performance win.

The optimizer's job — problem #9

The optimizer chooses among possible plans for the same query:

Query: Customers JOIN Orders

Possible plans:

  1. Scan customers → probe orders index
  2. Scan orders → hash join customers
  3. Sort both → merge join
Key insightOn what basis does the optimizer choose?＋

The optimizer picks based on estimated cost, considering:

  • Table sizes
  • Index availability
  • Data distribution
  • Memory constraints

Joins: signature strength & pain point — problem #10

SQL DB shines at joins, but they're computationally expensive.

Nested loop join

for each customer
   find matching orders

Hash join

build hash on smaller side
probe with larger side

Merge join

walk two sorted streams together
Tradeoff: each method has different performance characteristics — nested loops good for small left side, hash join good for memory, merge join good for pre-sorted data.
PROBLEM 11–14

Transactions: defining strength

▶

SQL DB is deeply about transactions.

BEGIN;
  debit A
  credit B
COMMIT;   -- or rollback on failure

ACID: the four pillars

ACIDMeaningKey insight
AtomicityAll or nothingNo partial updates
ConsistencyConstraints preservedDatabase enforces invariants
IsolationTransactions don't interfereConcurrency without corruption
DurabilityCommitted changes persistRecovery after crash

Concurrency control — problem #12

Two big families:

Locking — traditional, simple

Tx1 locks row R
Tx2 waits

MVCC — modern, efficient

row has versions
Tx1 sees old version
Tx2 sees new version after commit

Modern systems often use MVCC + locks where needed.

Isolation levels — problem #13

Not every system gives the same guarantees:

LevelGuaranteeTradeoff
Read UncommittedCan read uncommitted writesFast but unsafe
Read CommittedOnly see committed writesStandard default
Repeatable ReadSame reads within transactionMore consistent
SerializableComplete isolationMost safe but costly

The write-ahead log (WAL) — problem #14

The durability backbone:

update row
   |
   v
append change to WAL   <-- CRITICAL: log first
   |
   v
ack maybe after WAL durable
   |
   v
data page flushed later
Why it mattersWhat does the WAL let the database do after a crash?＋
  • Redo committed work (recovery)
  • Undo/ignore incomplete work (rollback)
PROBLEM 15–16

Memory & recovery

▶

Buffer manager — problem #15

Disk is slow, so SQL DB uses memory buffer pool for pages:

Disk pages <--> Buffer Pool <--> Query Execution

When query needs a page:

  1. Check buffer pool
  2. If miss → load from disk
  3. If hit → use directly
Key insightWhat is the buffer manager, essentially?＋
The buffer manager is essentially a cache for pages, and its efficiency determines query performance.

Crash recovery — problem #16

After a crash, recovery is non-negotiable:

Crash
  |
  v
restart
  |
  v
read checkpoint
  |
  v
redo committed WAL entries
  |
  v
undo incomplete work if needed
Mental modelDefine recovery in one line — then open.＋
Recovery = bring data pages back to a transactionally consistent state using log + checkpoints.
PROBLEM 17–19

Scaling & distribution

▶

Replication — problem #17

Classic SQL DB often starts as single primary, then adds replication:

Primary (writes)
  |
  +--> Replica 1 (reads)
  +--> Replica 2 (reads)

Replication often based on WAL/binlog shipping — primary writes log, replicas replay those changes.

Tradeoff: read scaling is easy; write scaling is hard.

Scaling tradeoffs — problem #18

Single-node SQL DB gives strongest simplicity, but scale creates tradeoffs:

PatternAdvantageProblem
Vertical scalingSimpleLimited, expensive
Read replicasGreat for readsWrites still bottleneck
PartitioningDistributing dataCross-partition queries
ShardingHorizontal scalingComplex, distributed transactions
Distributed SQLFull scalabilityCoordination overhead

Schema design — problem #19

The tension: logical correctness vs physical performance.

Normalization

  • Good: reduces duplication, preserves integrity
  • Bad: more joins, more overhead

Performance tricks

  • Denormalization
  • Materialized views
  • Careful indexing
Key insightWhere does schema design sit in the whole picture?＋
SQL DB is deeply tied to schema design — good relational design matters, but performance requires physical optimization.
PROBLEM 20

Putting it all together

▶

Example: orders query

SELECT *
FROM orders
WHERE customer_id = 1001
ORDER BY created_at DESC
LIMIT 20;

Strong design

  • Table: orders (heap or clustered)
  • Index: (customer_id, created_at DESC)

Execution likely

  1. Index seek to customer_id=1001
  2. Walk newest rows in index order
  3. Fetch top 20 (possibly from index alone if covering)
This is classic SQL strength: declarative query + efficient execution = powerful abstraction.

CORE

The core tension

▶

Every SQL database wrestles with this fundamental challenge:

How to provide strong correctness guarantees while achieving acceptable performance?

This tension manifests everywhere:

  • Indexes speed queries but slow writes
  • Transactions guarantee correctness but add overhead
  • Normalization preserves integrity but requires joins
  • Replication improves availability but adds complexity

The SQL DB is a study in tradeoffs: correctness vs performance, simplicity vs scalability, declarative ease vs implementation complexity.

CLOSING

Final thought

▶

Building a SQL database means mastering:

  1. The relational model (data organization)
  2. Storage (pages, indexes, buffers)
  3. Query optimization (plan selection, cost estimation)
  4. Transactions (ACID, concurrency, recovery)
  5. Distribution (replication, sharding, consensus)

Each layer presents deep intellectual challenges — and the magic is making it all work together seamlessly for the user who just sees:

App --> SELECT / INSERT / UPDATE / DELETE
That's the power of a SQL database.
